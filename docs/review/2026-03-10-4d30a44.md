## Structural Integrity

### 1. App.tsx Crosses 1,000 LOC — Web Client Orchestrator Growing
**Impact**: Compounds over time — every new feature adds state/callbacks here

`packages/web/src/client/App.tsx` is now 1,040 LOC, the largest file in the web package. It serves as the main application orchestrator handling:

| Section | Responsibility |
|---------|---------------|
| State management | 12+ `useState` hooks, 1 `useReducer` (ThreadState), 5+ `useRef` |
| RPC lifecycle | Connection state, reconnect logic, thread switching |
| Thread operations | Create, open, delete, resume, model/mode/effort changes |
| Input handling | Image upload, slash commands, steering queue |
| Modal management | Provider settings, tool settings, delete confirmation, connection |
| Effect hooks | 8+ `useEffect` blocks for side effects |

The file has grown steadily: it was not tracked in previous reviews because it was under the threshold. The growth is organic — each new user-facing feature (model selection per thread, image uploads, steering queue, connection modals) adds state variables and callbacks here because App.tsx is the single React component that owns all cross-cutting state.

Unlike server.ts (which had clear extraction targets — thread handlers, config handlers), App.tsx's growth is harder to decompose because React component state doesn't extract as cleanly as handler functions. The natural decomposition points are:
- **Thread operations** (~200 LOC): `openThread`, `deleteThread`, `startThread`, `resumeThread` → custom hook `useThreadManager()`
- **Connection/reconnect logic** (~100 LOC): reconnect state, modal, retry → custom hook `useConnectionMonitor()`

### 2. Web Server index.ts — +75 LOC from --log-file Flag
**Impact**: Cosmetic — justified growth for operational need

`packages/web/src/server/index.ts` grew from ~280 to 354 LOC (+74) from commit ffb4362, which adds `--log-file` flag support to mirror stdout/stderr to a file. This is operational infrastructure for desktop sidecar debugging — when the sidecar process runs headless, logs need to be captured to disk.

The growth is justified but the file is approaching the boundary where CLI argument parsing, server creation, and I/O routing should be separated. Not urgent at 354 LOC.

## Novel Perspective

### The Thirdparty Directory Is an Uncontracted Extension Surface

- **Why no one has seen this**: Previous reviews assessed the plugin system (catalog.ts, plugin-loader.ts) as an architectural feature of core. The `thirdparty/` directory was treated as "external code" — outside the review scope. But `thirdparty/overdare/` is not external. It ships inside the monorepo, is built by the desktop packaging pipeline (`apps/desktop/scripts/package.ts`), deployed by `init.rs`, and its `config.jsonc` and `systemprompt.txt` are loaded at runtime. It is first-party code living in a third-party directory.

- **Evidence**:
  - `thirdparty/overdare/config.jsonc` sets `defaultModel: "gpt-5.4"` (commit 4d30a44) — this is runtime configuration that overrides the core default model. It is loaded by `loadRuntimeConfig()` without any schema validation boundary between "core config" and "overdare config."
  - `thirdparty/overdare/systemprompt.txt` (10,754 bytes) is injected into the system prompt — it has the same authority as the core system prompt but lives outside `packages/core/`.
  - `apps/desktop/src-tauri/src/init.rs` (7,797 LOC) has `deploy_plugins()` that specifically handles `@overdare/` scoped packages (commit a7bb503), creating an implicit contract between the Rust init code and the TypeScript plugin directory structure.
  - `thirdparty/overdare/plugins/` contains 3 plugins totaling ~3,000+ LOC of TypeScript. These tools are discoverable by any session through the standard plugin pipeline — they have the same execution privileges as core tools (file system access, shell execution via bash tool calls they generate).
  - `.diligent/knowledge/knowledge.jsonl` (commit eadaf69) stores OVERDARE-specific domain knowledge that persists across sessions and influences agent behavior.

- **Implication**: The thirdparty directory has evolved from "example plugins" into a **product vertical** with its own model defaults, system prompt, domain knowledge, and 20+ specialized tools. But it has no contract with core — no schema for config overrides, no boundary for system prompt authority, no test coverage in the core test suite. If overdare's config.jsonc has an invalid field, it fails at runtime. If systemprompt.txt conflicts with the core system prompt instructions, the conflict is invisible. The current structure works because one developer maintains both. But it encodes an assumption: **the plugin author and the core author are the same person**. If that assumption becomes false (open-source plugins, multiple product verticals), the thirdparty directory needs a contract — validated config schemas, scoped system prompt injection with explicit priority, and a test harness that verifies plugin loading without running the full agent.
