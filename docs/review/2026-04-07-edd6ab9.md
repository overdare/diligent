### session/manager.ts: 723 LOC — growth warrants monitoring
**Impact**: Compounds over time

session/manager.ts grew from 659 LOC (2026-04-04) to 723 LOC. The +64 LOC growth comes from usage prefix comparison diagnostics (commit 62318a0), adding session-level state maps:
- `prevPromptHashesBySession` (Map keyed by sessionId)
- `currPromptHashesBySession` (Map keyed by sessionId)
- `promptSignatureCountBySession` (Map keyed by sessionId)
- `prevCacheReadBySession` (Map keyed by sessionId)

These maps grow with each session created but are only cleared via `resetUsageDebugState()` on session resume/create — not on session deletion. In a long-running web server process with many sessions, this represents unbounded memory growth. The feature is diagnostic-only (controlled by `DILIGENT_DEBUG_PROMPT_HASH`), so the risk is limited to debug scenarios, but the pattern of module-level maps keyed by sessionId without cleanup is a structural concern.

**Evidence**: `packages/runtime/src/session/manager.ts` lines 540-583

## Forward Compatibility

### `ProviderToolUseBlock.name` enum: extensibility ceiling approaching
**Horizon**: Breaks when a provider adds non-web native tools (e.g., code execution, file generation)

`ProviderToolUseBlockSchema.name` is `z.enum(["web_search", "web_fetch"])` (data-model.ts). When a new provider-native capability arrives, this enum must be extended, which changes the Zod schema union discriminant. All content block consumers (web `AssistantContentBlocks.tsx`, CLI `thread-store-utils.ts`, e2e `provider-native-blocks.test.ts`) that pattern-match on `name` will need updating. The `provider` field is also `z.enum(["openai", "chatgpt", "anthropic"])` — adding Google/Gemini web tools would require schema extension.

The cheaper fix: consider whether these should be `z.string()` with runtime validation, or keep as enums with an explicit extension protocol documented alongside P061.

**Evidence**: `packages/protocol/src/data-model.ts` ProviderToolUseBlockSchema definition

### Session-level debug state maps: unbounded growth in long-running processes
**Horizon**: Breaks when web server processes handle many sessions without restart

`session/manager.ts` maintains 4 `Map<string, ...>` objects keyed by sessionId for usage/prompt diagnostics. These are only cleared on session resume/create via `resetUsageDebugState()`, not on session deletion or cleanup. In the web server model (long-running process serving multiple clients), stale entries accumulate. The feature is gated behind `DILIGENT_DEBUG_PROMPT_HASH`, limiting the blast radius, but the pattern of process-lifetime maps keyed by ephemeral session IDs is architecturally fragile.

**Evidence**: `packages/runtime/src/session/manager.ts` lines 540-583

## Novel Perspective

### The Mega-Commit Pattern is Not Decomposition Failure — It's a Missing Staging Infrastructure

**Why no one has seen this**: The 2026-04-06 review identified mega-commits (46-file, 68-file) as a testing gap caused by insufficient decomposition. The recommendation was per-package commits with intermediate verification. This period's desktop promotion commit (72a102d: 286 files, 35,960 insertions) is even larger, yet it landed cleanly. The real issue is not developer discipline — it's that the repository lacks staging infrastructure that would make incremental commits the path of least resistance.

**Evidence**:

1. **The 286-file commit is structurally atomic**: Commit 72a102d renames `apps/desktop/` → `apps/overdare-agent/`, updates all workspace references, migrates plugins from `thirdparty/overdare/` into the app, and adds bundled bootstrap resources. Splitting this into per-package commits would create intermediate states where the workspace is broken — `package.json` references `apps/overdare-agent/` but the directory doesn't exist yet, or `thirdparty/overdare/` is deleted but the plugins haven't been copied yet. Git's rename detection works on whole-tree snapshots, so a split would also lose rename tracking.

2. **The CI workflows validate only the final state**: The release pipeline (`release.yml`, 363 LOC) and automation workflows (`passion-junior.yml`, `tech-lead.yml`) run on push/PR events, not on individual commits within a branch. There is no infrastructure to validate intermediate commit states during development. This means the cost of splitting a large change into sequential commits is pure overhead — each intermediate commit must be manually verified to not break the workspace, with no automated safety net.

3. **The monorepo workspace model rewards atomic changes**: Bun workspaces resolve packages by name through `package.json` `workspaces` entries. Renaming a package directory requires updating the workspace entry, the package's own `package.json` name/path, all tsconfig path aliases, all Makefile targets, and all CI workflow paths — simultaneously. The workspace model creates an all-or-nothing constraint that makes incremental migration commits actively harmful unless there's a deprecation/aliasing layer (which would be over-engineering for a rename).

4. **Contrast with the remediation PRs**: PRs #83, #85, #86, #87 each changed 1-3 files and addressed exactly one concern. These were naturally small because they modified behavior within existing package boundaries. The desktop promotion crossed every boundary — workspace config, CI, build scripts, packaging, Rust source, TypeScript tests, bootstrap resources. The difference isn't developer discipline; it's whether the change respects or crosses package boundaries.

5. **What would actually help**: A workspace-level `bun check` or `make verify` command that runs typecheck + test for all affected packages in dependency order, callable during development (not just in CI). Currently, the developer must manually run `bun test` in each affected package to verify intermediate states. A single command that validates workspace coherence would make splitting large changes into verifiable intermediate commits practical rather than burdensome.

**Implication**: Rather than prescribing commit decomposition as a process discipline, invest in a workspace-level verification command (`make check` or `bun run verify`) that validates typecheck + affected-package tests in dependency order. This would make incremental commits cheap to verify, naturally encouraging smaller commits without requiring manual coordination. The release pipeline already has the platform matrix; the missing piece is a developer-facing equivalent for local iteration.
