### 2. Shadow Protocol in rpc-bridge.ts — Structural Gap
**Impact**: Compounds when Desktop adds auth management UI

`rpc-bridge.ts` acts as both a transparent proxy for `DiligentAppServer` protocol messages AND a custom endpoint for 5 web-only methods that are not defined in `@diligent/protocol`:

| Method | Handler Location | Type Safety |
|--------|-----------------|-------------|
| `config/set` | rpc-bridge.ts:174 | `as { model?: string }` cast — no Zod validation |
| `auth/list` | rpc-bridge.ts:204 | Untyped — returns ad-hoc `{ providers, availableModels }` |
| `auth/set` | rpc-bridge.ts:216 | `as { provider?: string; apiKey?: string }` cast |
| `auth/remove` | rpc-bridge.ts:239 | `as { provider?: string }` cast |
| `auth/oauth/start` | rpc-bridge.ts:266 | No params needed |

The Web client calls these via `rpc.requestRaw()` (`packages/web/src/client/lib/auth-api.ts`, `use-provider-manager.ts`) — an escape hatch that bypasses typed dispatch.

This works today because only the Web client uses these methods. When Desktop adds its own auth management UI, it will either: (a) duplicate these handlers in the Tauri sidecar, or (b) use the same WebSocket path, which means the shadow protocol becomes a shared contract without formal types.

**Recommendation**: Formalize these 5 methods in `@diligent/protocol` with Zod schemas. This also brings the `ProviderName` cast at rpc-bridge.ts:219-220 and 242-243 under schema validation, eliminating the `as ProviderName` escape hatches.

## Cross-Package Coherence

### 8. Notification Adapter Extraction — High-Impact Win
**Scope**: packages/core, packages/cli, packages/web
**Impact**: Eliminates the #1 compounding risk from previous reviews

`ProtocolNotificationAdapter` in `packages/core/src/notification-adapter.ts` (~205 LOC) is now the single implementation for protocol notification → AgentEvent conversion. Both CLI (`rpc-client.ts`, which dropped from ~270 LOC to 81 LOC) and Web (`thread-store.ts`, which dropped from ~420 LOC to 494 LOC but with a cleaner `reduceAgentEvent` function) consume it.

This means adding a new notification type now requires:
1. Add schema to `@diligent/protocol`
2. Add handler in `ProtocolNotificationAdapter`
3. Both frontends automatically receive the new events

Previously: 2 independent handlers needed manual sync. This is the most impactful sustainability improvement since the protocol package was introduced.

### 9. ModelInfo Type Still Local to Web — Persistent
**Scope**: packages/web
**Impact**: Compounds when Desktop adds model selection UI
