## Novel Perspective

### Extractions Are Copying Behavior, Not Moving Abstractions

- **Why no one has seen this**: Previous reviews focused on whether extractions happen and whether LOC decreases. The quality metric was "did App.tsx shrink?" (yes, -89 LOC). No review has examined whether the extraction preserved, duplicated, or degraded the abstractions that existed in the source file.

- **Evidence**: The `useThreadManager()` extraction (3d08e30) moved thread CRUD operations out of App.tsx. But the move duplicated logic instead of transplanting it:

  1. `pushThreadUrl()` was a helper function in App.tsx (6 LOC) that encapsulated URL update logic: check if the current URL differs, then pushState. The extraction inlined `window.history.pushState()` directly in two places in use-thread-manager.ts (lines 83, 125) instead of moving the helper. The original function was then renamed to `_pushThreadUrl` (429d784) to suppress the unused-variable lint warning ‚Äî creating dead code rather than deleting it.

  2. `getThreadIdFromUrl()` (App.tsx:153) remains in App.tsx as a standalone function that extracts `threadId` from `window.location.pathname`. This is the companion to `pushThreadUrl` ‚Äî one reads the URL, the other writes it. The extraction split these URL-management concerns: the reader stayed in App.tsx, the writer's logic was duplicated into the hook, and the writer's function became dead code.

  This is not a quality problem with this specific extraction ‚Äî it's a structural pattern of how copy-based refactoring works. When code is extracted by copying relevant sections into a new file (as AI-assisted refactoring tends to do), the result is a **behavior duplicate** rather than an **abstraction transfer**. The new file works correctly (the hook pushes URLs in the right places), but the old file accumulates dead code and the shared utility that should exist (`thread-url.ts` with `getThreadIdFromUrl` + `pushThreadUrl`) is never created.

  The cost is small today (6 LOC of dead code). But as the review process continues to recommend extractions ‚Äî `useConnectionMonitor()`, `useImageUpload()`, and eventually SessionManager decomposition ‚Äî each extraction that copies behavior instead of moving abstractions will leave behind dead code, duplicated logic, and missed utility-extraction opportunities. The fix is a post-extraction checklist: after every hook extraction, grep for functions in the source file that are no longer called and either move or delete them.

- **Implication**: Add a post-extraction verification step: after extracting a hook or handler file, run `bun lint` on the source file and treat any new unused-variable warnings as indicators of incomplete extraction (delete or move), not as noise to suppress with `_` prefixes.
