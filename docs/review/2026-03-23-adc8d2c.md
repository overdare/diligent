## Structural Integrity

### COLLAB_TOOL_NAMES defined in 3 separate locations — centralization incomplete
**Impact**: Compounds with each new collab tool

The tool-metadata registry (c69b1ec) was created to be the single source of truth for tool capability sets. `COLLAB_TOOL_NAMES` is correctly derived from the registry in `tools/tool-metadata.ts:62`. However, two additional hardcoded definitions exist:

1. **`packages/runtime/src/agent/agent-types.ts:59`**: `new Set(["spawn_agent", "wait", "send_input", "close_agent"])` — introduced in the same commit (adc8d2c) that the registry was already available. This set is used by `formatSpawnAgentToolDescription()` to filter collab tools from agent type descriptions.

2. **`packages/cli/src/tui/components/thread-store-utils.ts:9`**: `new Set(["spawn_agent", "wait", "send_input", "close_agent"])` — used for collab tool rendering in the TUI. This predates the registry and was not migrated.

If a 5th collab tool is added (e.g., `transfer_agent`), it must be updated in 3 files instead of 1. The `collab/registry.ts:11` correctly imports from `tool-metadata` — the other two locations should do the same.

**Evidence**: `packages/runtime/src/agent/agent-types.ts:59`, `packages/cli/src/tui/components/thread-store-utils.ts:9`, `packages/runtime/src/tools/tool-metadata.ts:62`

## Novel Perspective

### The Custom Agent System Creates a Shadow Type System for Tool Capabilities That Will Diverge from the Registry

**Why no one has seen this**: The custom agents feature (adc8d2c) was just committed. Previous reviews focused on tool-metadata centralization and type duplication. The novel angle is that custom agents introduce a *second representation* of tool access control that lives outside the code — in AGENT.md frontmatter files — and this representation has fundamentally different lifecycle and validation properties than the in-code registry.

**Evidence**:
1. **AGENT.md frontmatter is a user-authored declaration**: `.diligent/agents/code-reviewer/AGENT.md` contains `tools: read, glob, grep, ls`. This is a string list parsed at discovery time (`frontmatter.ts:11-16`). It's validated against `TOOL_CAPABILITIES` (`frontmatter.ts:21`), but only at agent load time — not at spawn time.

2. **The in-code registry is developer-authored and compile-time checked**: `TOOL_CAPABILITIES` in `tool-metadata.ts` is a TypeScript `Record<string, ToolCapabilities>`. Adding or removing a tool is a code change that goes through typecheck, tests, and review.

3. **The two representations have different change velocities**: When a built-in tool is renamed (e.g., `edit` → `apply_patch` in commit 404be3a), the code registry updates atomically. But AGENT.md files in user projects are not updated — they still reference the old tool name. The validation catches this at load time, but the error message goes to `AgentLoadResult.errors[]`, which is currently only surfaced in the system prompt (not as a user-visible warning).

4. **Per-spawn `allowed_tools` adds a third representation**: The `spawn_agent` tool accepts `allowed_tools: string[]` from the LLM (spawn-agent.ts:30-35). This is validated at spawn time by intersection with the parent tool list — but the LLM can hallucinate tool names that don't exist, and the intersection silently drops them (the child agent just gets fewer tools than intended, with no error).

5. **The layered narrowing in `registry.ts:108-119` applies 4 filters in sequence** — collab exclusion, readonly filter, agent-definition allowedTools, and per-spawn allowedTools. Each is a `Set` intersection. If any layer has a stale or misspelled tool name, the child agent silently loses access to that tool. There is no "expected tools vs actual tools" reconciliation.

**Implication**: The tool access control system now has three layers of tool-name-as-string references: the TOOL_CAPABILITIES registry (authoritative), AGENT.md frontmatter (user-authored), and LLM-generated spawn parameters (ephemeral). Only the first is validated at compile time. The system needs a reconciliation step — either at agent load time (warn when frontmatter tools don't match the current registry) or at spawn time (warn when allowed_tools narrows to an empty set or loses tools). Without this, a renamed tool silently degrades agent capabilities with no visible error.
