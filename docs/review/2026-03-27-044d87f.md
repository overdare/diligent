## Structural Integrity

### Global agent depth limit — 3-review persistent issue now resolved
**Impact**: Positive — eliminates unbounded recursive agent spawn

PR #57 (7299793) adds a `depth` field to `CollabToolDeps` (default 3), stored as an instance field in `AgentRegistry` (`registry.ts:92,97`). The spawn method checks `this.depth <= 0` before allowing spawn (`registry.ts:148-150`) and passes `depth: this.depth - 1` to child deps (`registry.ts:217`). The `depth` field is excluded from `MutableCollabDeps` (`types.ts:85`), preventing accidental reset via `updateDeps()`.

With `maxAgents=8` and `depth=3`: the theoretical maximum concurrent sessions is now bounded at 8+64+512=584, but in practice depth=3 means at most 3 levels of nesting. The previous concern about `maxAgents^depth` unbounded growth is structurally eliminated.

**Evidence**: `packages/runtime/src/collab/registry.ts:92,97,148-150,217`, `packages/runtime/src/collab/types.ts:55-59,85`, `packages/runtime/test/collab/registry.test.ts` (2 new depth tests)

## Cross-Package Coherence

### E2e WS factory duplicates web server's createWsPeer — intentional but undocumented
**Scope**: packages/e2e, packages/web

The `ws-server-factory.ts` in e2e replicates the `createWsPeer` implementation from `packages/web/src/server/index.ts:165-194`. Both create `RpcPeer` objects with identical message parsing logic (JSONRPCMessageSchema.parse), listener management, and error handling. The duplication is structurally intentional — e2e should not import web server internals — but the two implementations must stay in sync. A transport-level serialization change in the web server that isn't mirrored in the e2e factory would create false-positive test results.

The connection ID format differs (`web-${UUID}` vs `ws-test-${counter}`), confirming the duplication is deliberate. However, no comment in either file references the other.

**Evidence**: `packages/e2e/helpers/ws-server-factory.ts:58-87`, `packages/web/src/server/index.ts:165-194`

## Novel Perspective

### The E2E WebSocket Factory Creates a Shadow Transport Contract That Drifts Independently

**Why no one has seen this**: The WebSocket e2e test (PR #58) was reviewed and merged as a valuable addition to transport coverage — which it is. Previous reviews tracked the *absence* of transport-level tests. Now that the test exists, the structural consequence becomes visible: the e2e test validates a *copy* of the transport implementation, not the actual production transport.

**Evidence**:

1. **`packages/web/src/server/index.ts:165-194`** defines the production `createWsPeer` function. It handles WebSocket message parsing, `JSONRPCMessageSchema` validation, listener registration, and error response serialization.

2. **`packages/e2e/helpers/ws-server-factory.ts:58-87`** defines an independent `createWsPeer`-equivalent implementation. It uses the same `JSONRPCMessageSchema.parse()` call, the same listener-based message dispatch, and the same `JSON.stringify` serialization. But it is a separate implementation, not an import.

3. **Neither file references the other.** There is no comment, test assertion, or structural mechanism ensuring the two stay in sync.

4. **The divergence risk is specific and measurable.** If the web server's `createWsPeer` adds:
   - Binary message support (ArrayBuffer handling)
   - Message batching or compression
   - Connection-level authentication headers
   - Graceful close with status codes

   The e2e factory would continue using the old behavior. Tests would pass against the shadow transport, not the production one.

5. **The 2026-03-26 review's Forward Compatibility finding** ("E2E tests use in-memory RPC only") is now partially resolved — real WebSocket transport is tested. But the resolution introduced a new, subtler risk: the transport tested is a *faithful-but-independent* replica, not the production code path.

**Implication**: Extract the `createWsPeer` / RPC-peer-over-WebSocket logic into a shared utility in `packages/runtime/src/rpc/` (where other RPC channel utilities already live). Both `packages/web/src/server/index.ts` and `packages/e2e/helpers/ws-server-factory.ts` should import from this shared implementation. This ensures the e2e test exercises the actual production transport code, not a look-alike. The extraction is small (~30 LOC) and the import chain (`runtime → web`, `runtime → e2e`) already exists for other utilities.
