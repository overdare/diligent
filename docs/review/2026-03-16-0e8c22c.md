## Structural Integrity

### 1. Tool Render Payload Refactoring — Producer-First Architecture
**Impact**: Positive — eliminates the largest persistent issue (tool-info.ts shadow registry)

The render payload refactoring (5888108 + b6f8e98) is the defining architectural change of this period. Previously, tool render payloads were derived on the consumer side: web's `tool-info.ts` (513 LOC) contained `deriveRenderPayload()` which inspected raw tool input/output text to reconstruct structured rendering data. TUI's `transcript-store.ts` had a parallel derivation path.

**After the refactoring:**
- Tools produce `ToolRenderPayload` at execution time in runtime (`render-payload.ts`: 515 LOC)
- Payloads flow through protocol as part of `item/ended` notification
- Consumers (web, TUI) receive pre-computed render blocks and just render them
- `tool-info.ts` dropped from 513→102 LOC (-80%), with `deriveRenderPayload()` becoming a 1-line delegation to `createTextRenderPayload()` from `@diligent/runtime/tools`

This resolves the "shadow tool registry" persistent issue tracked since 2026-03-09 (8 consecutive reviews). The pattern is now correct: **producers know how to describe their output, consumers know how to render descriptions**.

## Novel Perspective

### The Render Payload Refactoring Exposed a Hidden Coupling Pattern: "Protocol Enrichment at the Wrong Layer"

- **Why no one has seen this**: Previous reviews tracked tool-info.ts as a "shadow registry" — a consumer-side problem. The render payload refactoring fixed that specific instance by moving rendering to producers. But the fix reveals a broader pattern: **protocol messages that arrive "too thin" force consumers to re-derive information that was available at the producing layer**. tool-info.ts was the most visible symptom, but the pattern exists elsewhere.

- **Evidence**:
  1. **Transcript hydration**: `hydrateThreadHistory()` in `app-session-lifecycle.ts:151-189` casts content blocks to extract text: `(b as { text: string }).text` (line 169), `(b as { thinking: string }).thinking` (line 174). These casts exist because `TranscriptEntry.message.content` is typed as a generic content block array, and the consumer must inspect and cast to extract displayable content. The protocol carries the raw provider message format — the consumer must re-derive the display text.

  2. **thread-hydration.ts** (429 LOC, web): This entire file exists because `ThreadReadResponse.transcript` carries raw `TranscriptEntry` objects that must be parsed into renderable state. The protocol transmits "what happened" (provider messages with content blocks), not "what to display" (parsed text, tool results, thinking summaries). Every consumer must independently build the display model from the raw transcript.

  3. **event-mapper.ts** (318 LOC, runtime): Maps `AgentEvent` to `DiligentServerNotification`. This is the "enrichment layer" — it already exists for tool render payloads (added in the refactoring). But it only enriches `item/ended` notifications with `ToolRenderPayload`. Other notifications (message deltas, thinking blocks) pass through as-is, leaving consumers to handle raw content blocks.

  4. **The render payload refactoring proved the fix works**: Before the refactoring, tool output was "too thin" in the protocol (just text). After, it's enriched with structured blocks. Consumers went from 513 LOC of derivation to 1-line delegation. **The same enrichment pattern could apply to transcript entries**: instead of transmitting raw provider messages in `ThreadReadResponse`, the server could transmit pre-parsed `DisplayEntry` objects with extracted text, thinking summaries, and tool result summaries.

  The pattern is: **when the protocol transmits raw internal representations, every consumer independently builds a display model. When the protocol transmits pre-computed display representations, consumers just render.** The tool render payload refactoring moved one data path from the first pattern to the second. The transcript hydration path remains in the first pattern — with two independent parsers (39 LOC + 429 LOC) deriving display state from raw entries.

- **Implication**: The shared transcript parser recommended in 4 consecutive reviews is the right direction but addresses the symptom. The root fix is **server-side transcript enrichment**: when `thread/read` is called, the server should return pre-parsed display entries (text extracted, thinking summarized, tool results with render payloads) instead of raw `TranscriptEntry` objects. This would:
  - Eliminate both consumer-side parsers entirely (39 + 429 = 468 LOC)
  - Follow the same pattern that resolved tool-info.ts
  - Make the TUI-Web transcript divergence structurally impossible — there would be nothing to diverge on

  The cost is a `TranscriptDisplayEntry` type in protocol and an enrichment function in the server's `thread/read` handler. The payoff is eliminating the most persistent cross-package coherence issue in the project's history.
