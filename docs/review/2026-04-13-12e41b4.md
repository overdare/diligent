## Forward Compatibility

### Shared Reducer Incompleteness Blocks Web Adoption
**Horizon**: Breaks when Web client attempts adoption

`packages/protocol/src/conversation-reducer.ts` handles ~10 event types but is missing:
- Collab events: `collab_spawn`, `collab_wait`, `collab_close`, `collab_interaction_begin`, `collab_interaction_end`
- Provider-native content blocks within `message_delta` (thinking blocks, tool-use blocks)
- Knowledge and usage event accumulation
- Thinking-duration tracking (Web tracks `thinkingStartTime` / `thinkingDone`)

Web's `thread-store.ts` (lines ~200-648) handles all of these. Until the protocol reducer covers them, Web cannot adopt without losing functionality. This is the single blocking dependency for completing reducer convergence.

### SessionManager State Field Accumulation
**Horizon**: Breaks when multi-session or session-migration features arrive

`packages/runtime/src/session/manager.ts` carries 4 private Maps for prompt deduplication and cache tracking (lines 66-69: `prevCacheReadBySession`, `prevPromptHashesBySession`, `currPromptHashesBySession`, `promptSignatureCountBySession`). These Maps are keyed by sessionId and never cleaned up in long-running processes. While gated behind debug instrumentation, the pattern creates unbounded memory growth proportional to session count × history depth.

### Thread-Handlers Model/Effort Resolution Scans Full Session History
**Horizon**: Performance degrades with session count

The newly added `getLatestModelFromSessions()` and `getLatestEffortFromSessions()` (thread-handlers.ts lines ~648-716) iterate through all sessions for the current cwd, reading and parsing session files to find the most recent model/effort. For users with many sessions, this performs unbounded I/O on every new thread creation. A simple cache or last-used-model persistence would eliminate this.

## Novel Perspective

### The Protocol Reducer Is Converging Toward a Framework — But Has No Test Suite

- **Why no one has seen this**: Previous reviews focused on *adoption* of the shared reducer (who calls it?) and *completeness* (which events does it handle?). No review has examined whether the reducer itself is tested.
- **Evidence**: `packages/protocol/src/conversation-reducer.ts` (110 LOC) has zero test files. The protocol package's test directory does not contain a `conversation-reducer.test.ts`. Meanwhile, the downstream consumers (Web's thread-store has `packages/web/test/client/lib/thread-store.test.ts`, CLI has no reducer test) test their own independent implementations.
- **Implication**: As the shared reducer grows to handle collab events, provider-native content, and thinking-duration tracking (required for Web adoption), it becomes the single source of truth for all clients' state behavior. An untested shared reducer that all three clients delegate to creates a higher-blast-radius regression surface than three independent reducers that each have (or lack) their own tests. Before expanding the reducer for Web adoption, it needs a comprehensive test suite covering all event types — ideally derived from the existing Web/CLI test cases that exercise the same event handling independently.
