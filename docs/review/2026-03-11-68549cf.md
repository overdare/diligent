## Novel Perspective

### The Fix-After-Ship Pattern Is a Missing Feedback Loop, Not a Quality Problem

- **Why no one has seen this**: Previous reviews track rework ratio as a velocity metric — "what fraction of commits are rework." This period's 28% rework ratio looks like a quality signal ("features ship with bugs"). But the pattern reveals something structural: the fix commits are not random. They cluster specifically around **cross-platform behavior** (Windows OAuth, Windows paths, Windows browser open) and **state coordination** (effort reset on model switch, config modal close, plan panel scroll). These are categories where unit tests cannot catch the failure because the failure mode is environmental or interaction-based.

- **Evidence**:
  - 5 of 13 fix commits target Windows-specific behavior (`31e9630`, `c4253c1`, `0f174ef`, `0ee133d`, `8141fc6`). The project has no Windows CI runner — these are discovered by manual testing on a Windows desktop build.
  - 4 of 13 fix commits target UI state coordination (`e095fd2`, `d43efd4`, `1554c89`, `8cf963c`). These are interaction-order bugs (modal close timing, scroll position, effort label sync) that require the full web app running to reproduce.
  - The `packages/e2e/` package exists but tests only the protocol layer. There are no UI integration tests that render the React app and simulate user interactions.
  - `packages/web/test/components.test.tsx` exists and has grown this period (+68 LOC in 68549cf, +10 LOC in dd3f0e2, +47 LOC in a56dc30), but it tests component rendering in isolation, not state coordination between App.tsx and its children.

- **Implication**: The 28% rework ratio is not solvable by "writing better code" or "more careful review." It's a **testing surface gap**: the project has strong unit tests (1,085+) and strong protocol tests (e2e/), but no integration tests that exercise the two failure modes that generate the most fixes — cross-platform runtime behavior and multi-component state coordination. The fix is not to add Playwright or Cypress (heavy, flaky, not worth the maintenance for a solo developer). The fix is narrower: (1) a Windows CI runner in GitHub Actions that at minimum runs `bun test` and `bun run typecheck` on Windows, catching platform-specific type and runtime issues before they reach manual testing; (2) a `web/test/integration.test.tsx` that mounts App.tsx with a mock RPC client and simulates the state transitions that generate fix commits (model switch → effort reset, modal open → close → reopen, thread switch → state cleanup).
