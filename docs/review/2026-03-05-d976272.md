## Structural Integrity

### 1. Tool Catalog System (P032) — Major New Extension Point, Well-Isolated
**Impact**: Positive — foundational for plugin ecosystem

Commit 5416388 adds a complete tool resolution pipeline: builtin toggles, plugin loading, conflict resolution, and immutable tool enforcement. The design is clean:

| Module | LOC | Responsibility |
|--------|-----|---------------|
| `tools/catalog.ts` | 141 | Resolution pipeline — merge builtins + plugins, apply config |
| `tools/plugin-loader.ts` | 108 | Dynamic import with manifest validation |
| `tools/immutable.ts` | 8 | Force-enable list (plan, request_user_input) |
| `config/schema.ts` (additions) | 17 | Zod schema for `tools` config section |

Key design properties:
- **Pure pipeline**: `buildToolCatalog(builtins, config, cwd)` → `ToolCatalogResult` — no global state
- **Conflict policy**: `error` (default, log and skip), `builtin_wins` (silent skip), `plugin_wins` (replace)
- **Immutable enforcement**: `plan` and `request_user_input` cannot be disabled regardless of config
- **Async support**: `buildDefaultTools()` is now async, `SessionManager.resolveAgentConfig()` preserves sync path for steering timing

The `buildAgentConfig` factory in `DiligentAppServerConfig` and `SessionManagerConfig.agentConfig` both now support `Promise` returns, cleanly propagated through the sync/async resolution in `SessionManager.resolveAgentConfig()`. The sync path is critical for steering message delivery timing — the code explicitly documents this (manager.ts:251-253).

Tests: 10 catalog tests + 9 plugin-loader tests covering builtin toggles, immutable enforcement, conflict policies, and plugin loading scenarios.

### 2. Parallel Tool Execution (D015) — Clean Implementation
**Impact**: Positive — performance improvement with bounded complexity

Commit af8793d adds parallel execution to the agent loop. The design follows an all-or-nothing pattern:

```typescript
const allParallel = toolCalls.length > 1 && toolCalls.every(tc => registry.get(tc.name)?.supportParallel);
```

When all tools in a turn support parallel (read, glob, grep, ls, plan), they execute via `Promise.all`. Otherwise, sequential execution (the original path) is used. Key properties:
- **Opt-in per tool**: `supportParallel?: boolean` on the `Tool` interface — plugins can opt in
- **Extracted helper**: `buildToolContext()` eliminates code duplication between parallel and sequential paths
- **Event ordering**: parallel emits all `tool_start` upfront, then results in original order
- **4 comprehensive tests**: parallel, mixed (falls back to sequential), single tool, default-sequential

The `loop.ts` grew from ~397 LOC to 517 LOC, but the added complexity is justified by the extracted helper and clean parallel/sequential branching.

## Forward Compatibility

### 14. Plugin System (P032 Phase 1) — Foundation Laid
**Horizon**: Plugin ecosystem readiness

The tool catalog system establishes the extension point for third-party tools:
- `DiligentConfig.tools.plugins` accepts `{ package: string, enabled?: boolean, tools?: Record<string, boolean> }[]`
- `loadPlugin()` dynamically imports npm packages and validates `DiligentToolPlugin` manifests
- Conflict resolution ensures builtins aren't silently replaced

This is Phase 1 only — `loadPlugin()` currently imports from the cwd's `node_modules`. Future phases will need:
- Plugin discovery (scanning `package.json` dependencies)
- Plugin sandboxing (currently plugins run in the same process)
- Plugin versioning (no manifest version field yet)

These are natural extensions, not architectural blockers.

### 15. Provider Registry — Still If/Else Chain
**Horizon**: Adding 4th provider

`provider-manager.ts:179-185` remains:
```typescript
if (provider === "openai") {
  stream = createOpenAIStream(apiKey, this.baseUrls.openai);
} else if (provider === "gemini") {
  stream = createGeminiStream(apiKey, this.baseUrls.gemini);
} else {
  stream = createAnthropicStream(apiKey);

```

The `else` fallback silently defaults to Anthropic for unknown providers. TypeScript catches omissions in `DEFAULT_MODELS` and `PROVIDER_HINTS` (both `Record<ProviderName, ...>`), but `getOrCreateStream()` is the one location that silently falls through.

Now that the plugin system introduces a pattern for tool registry lookups, this if/else could follow the same declarative pattern — but the urgency is low with 3 providers.
