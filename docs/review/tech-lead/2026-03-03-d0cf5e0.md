## Structural Integrity

### 1. Protocol Package as Single Source of Truth — Achieved
**Impact**: Foundation — enables all cross-package coherence

`packages/protocol/` now defines 11 client request methods, 1 client notification, 14 server notifications, and 2 server request methods — all with Zod schemas. Both CLI and Web import types from `@diligent/protocol` for their RPC layers. The `DiligentAppServer` in core imports protocol types for its JSON-RPC dispatch boundary.

The protocol package correctly includes `account/login/completed` and `account/updated` notification schemas (`server-notifications.ts:130-146`), added as part of the P015 auth notification refactor. Methods are registered in `DILIGENT_SERVER_NOTIFICATION_METHODS` (`methods.ts:39-40`).

### 2. Core Types vs Protocol Types — Intentional Duplication, Manageable Risk
**Impact**: Compounds over time — each new event type requires sync in two places

Core defines ~15 types as plain TypeScript interfaces (`types.ts`, `agent/types.ts`, `tool/types.ts`): `Message`, `AgentEvent`, `Usage`, `ApprovalRequest`, etc. Protocol defines structurally identical types as Zod schemas (`data-model.ts`). The `app-server/server.ts` bridges the two at the JSON-RPC boundary.

This is an **intentional architectural choice** — core avoids Zod on the hot path, protocol validates at the wire boundary. The risk is structural drift: if core adds a field to `AgentEvent` but protocol doesn't, the notification will silently drop it. Currently the two are in sync. The mitigation is the existing test suite (764 tests), which exercises both sides.

**Recommendation**: Acceptable as-is. If a drift bug occurs, consider generating core types from protocol Zod schemas. Don't pre-optimize.

### 3. ProviderName Triple Definition — Compounding Slowly
**Impact**: Compounds with each new provider (next: likely DeepSeek or local models)

`ProviderName` is defined independently in three locations:
- `packages/protocol/src/data-model.ts:305` — `ProviderNameSchema = z.enum([...])` (canonical)
- `packages/core/src/provider/provider-manager.ts:13` — `type ProviderName = "anthropic" | "openai" | "gemini"`
- `packages/core/src/auth/auth-store.ts:8` — `type ProviderName = "anthropic" | "openai" | "gemini"`

Adding a 4th provider requires manually updating all 3 locations. The `index.ts` re-export already aliases one as `AuthProviderName` to avoid collision — a sign the duplication causes friction.

**Fix cost now**: 15 minutes — import from protocol in both core files, or derive from `PROVIDER_NAMES` array const.
**Fix cost after 4th provider**: debugging session when one location is missed.

## Cross-Package Coherence

### 9. `model.provider ?? "anthropic"` — Scattered Default (10+ occurrences)
**Scope**: packages/cli (8 files), packages/core (1 file)
**Impact**: Compounds with each provider — "anthropic" as implicit default is never centralized

Occurrences in CLI alone:
- `config.ts:38`, `index.ts:58`, `app.ts:154,229`
- `model.ts:18,43,77`, `provider.ts:56,202`
- `provider-manager.ts:101` (core)

The fallback `"anthropic"` should be a `DEFAULT_PROVIDER` constant exported from `@diligent/core`. Currently, if the default provider changes (e.g., to a local model), every occurrence must be found manually.

**Fix cost now**: 20 minutes — extract constant, find-replace.
**Fix cost later**: Each new entry point (desktop CLI, API server) copies the pattern.

### 10. Notification Method Constants vs Raw Strings — Inconsistent
**Scope**: packages/cli, packages/web
**Impact**: Cosmetic but pattern-setting

`runner.ts` correctly uses `DILIGENT_SERVER_NOTIFICATION_METHODS.TURN_COMPLETED` constants from protocol. But `app.ts:547,555` and `thread-store.ts` (web) use raw string literals like `"turn/completed"`, `"item/started"`, etc. Both approaches work, but the inconsistency means new code may follow either pattern.

**Fix cost**: 15 minutes per package — import and use constants consistently.

### 11. ModelInfo Type Still Local to Web
**Scope**: packages/web
**Impact**: Blocks any future consumer of model info (desktop native UI, API)

`ws-protocol.ts:46-54` defines `ModelInfo` as a hand-maintained projection of core's `Model` type. Core `Model` has `defaultBudgetTokens`, `supportsThinking`; `ModelInfo` has `supportsThinking` but not `defaultBudgetTokens`. Core `Model` has `provider` as optional; `ModelInfo` has it as required string.

**Fix cost now**: 30 minutes — create `ModelInfo` schema in `@diligent/protocol`.
**Fix cost after desktop adds model UI**: Two separate ModelInfo definitions to maintain.

### 13. E2E Tests Bypass RPC Protocol
**Scope**: packages/e2e
**Impact**: Protocol-level regressions are caught only by unit tests in packages/web

`packages/e2e/conversation.test.ts` tests `agentLoop` directly — it never exercises the JSON-RPC WebSocket path. Adding/renaming a protocol method could break Web/CLI without e2e catching it. The `packages/web/test/` directory has good unit coverage for the bridge, partially mitigating this.

**Recommendation**: Add one smoke-test in e2e that starts a WebSocket server and runs a minimal turn through the full path. Not urgent — unit tests provide reasonable coverage.
