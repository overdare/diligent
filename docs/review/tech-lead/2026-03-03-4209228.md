## Structural Integrity

### 1. ProviderName Triple Definition — Compounding, Unchanged
**Impact**: Compounds with each new provider

`ProviderName` remains independently defined in 3 locations:
- `packages/protocol/src/data-model.ts:305` — `ProviderNameSchema = z.enum([...])`
- `packages/core/src/provider/provider-manager.ts:13` — `type ProviderName = "anthropic" | "openai" | "gemini"`
- `packages/core/src/auth/auth-store.ts:8` — `type ProviderName = "anthropic" | "openai" | "gemini"`

The `index.ts` re-export aliases one as `AuthProviderName` to avoid collision — a friction symptom. Adding a 4th provider requires manually updating all 3 locations plus `DEFAULT_MODELS`, `PROVIDER_HINTS`, `AuthKeys` interface, and `getOrCreateStream()`. That's 6 edits across 3 files for what should be a 1-file registry addition.

**Fix cost now**: 15 minutes — import from protocol or derive from `PROVIDER_NAMES` const.
**Fix cost after 4th provider**: debugging session when one location is missed.

## Cross-Package Coherence

### 8. Notification Handling Duplication — Two Parallel Implementations
**Impact**: Compounds with each new notification type; highest cross-package risk
**Scope**: packages/cli + packages/web

Two independent implementations convert protocol notifications to display state:

| Concern | CLI (`ProtocolNotificationAdapter`) | Web (`reduceServerNotification`) |
|---------|-------------------------------------|----------------------------------|
| File | `cli/src/tui/rpc-client.ts:95-269` | `web/src/client/lib/thread-store.ts:160-417` |
| State model | `Map<itemId, AssistantMessage/ToolCall>` | `RenderItem[]` in reducer |
| Output | `AgentEvent` stream | `RenderItem` dispatch |
| LOC | ~175 | ~260 |

Both independently implement:
- Item started → create slot (CLI lines 195-249, Web lines 163-194)
- Item delta → accumulate text/thinking (CLI lines 254-294, Web lines 196-235)
- Item completed → finalize (CLI lines 299-348, Web lines 237-268)

Adding a new notification type requires updating both. They handle different notification subsets today:

| Notification | CLI | Web |
|---|---|---|
| `steering/injected` | Not handled | Converts to user items |
| `loop/detected` | Not handled | Toast |
| `knowledge/saved` | Not handled | Toast |
| `account/*` | Not handled | Provider manager callbacks |

This divergence means the TUI and Web provide different user experiences for the same protocol.

**Fix**: Extract a shared `ProtocolNotificationReducer` in `@diligent/core` that both frontends wrap with their rendering logic. This is the single highest-impact refactor for sustainability.

## Summary

The project moves from GREEN to YELLOW — not because things got worse, but because the previous review's actionable items (all flagged as "15-minute fixes") have accumulated without remediation. The 11 commits since the last review were all valuable infrastructure work (EXPLORE.md, CI, lefthook), but zero source code fixes were applied. The individual items are still small, but the pattern of deferring cleanup while shipping tooling creates a gap between the project's strong architectural intent and its actual codebase state.

The deepest new finding is the **notification handling duplication** (Finding #8): two independent 175-260 LOC implementations in CLI and Web that convert the same protocol notifications into different display models. This is the highest-compounding risk because every new notification type requires changes in both, and the two already diverge on which notifications they handle. Fixing this before MCP or Approval work lands will save significant effort.

Test suite health is excellent (795/0/1751). The EXPLORE.md navigation system is a meaningful DX improvement. Decision documentation (88 entries) remains comprehensive. The project has strong foundations — the YELLOW is a call to invest one focused session in the cleanup queue before resuming feature work.
