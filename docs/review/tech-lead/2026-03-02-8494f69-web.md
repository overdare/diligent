## Sustainability Verdict

**YELLOW — Clean architecture and passing tests, but one UX regression and three type-synchronization points that compound with each new provider or protocol mode.**

The RpcBridge + thread-store reducer + WebRpcClient triad is well-designed: the protocol boundary is clear, the reducer is pure and testable, reconnect logic is explicit. Two issues prevent GREEN: (1) WebSocket reconnection creates a new thread instead of resuming the previous one — users lose context on any network hiccup; (2) three small type-sync points (`ModelInfo`, `ProviderName`, `Mode` literal) require manual parallel updates and will compound as providers/modes grow.

## Structural Integrity

### Architecture: Clean 3-Layer Separation

Import chain is disciplined:

```
client/ → @diligent/protocol (types only), shared/ws-protocol
server/ → @diligent/core, @diligent/protocol, shared/ws-protocol
shared/ → @diligent/protocol (types only)
```

No circular imports. Client never touches `@diligent/core`. Server never touches React. The `shared/ws-protocol.ts` layer correctly mediates between the core protocol and the WebSocket transport format.

**Impact**: None. Layer boundary is structurally sound.

### `ModelInfo` Duplicates Core's `Model` Type

`ws-protocol.ts:45-53` defines `ModelInfo` as a hand-mapped subset of `@diligent/core`'s `Model`. In `index.ts:68-76`, KNOWN_MODELS is explicitly mapped to this type field by field. When a new field is added to `Model` (e.g., `supportsExtendedOutput` for a future provider), it must be added to (1) `Model` in core, (2) `ModelInfo` in ws-protocol, and (3) the mapping in `index.ts`.

The fix is trivial: re-export a `ModelInfo` type from `@diligent/protocol` so all three packages share one definition.

**Impact**: Compounds over time (one new field → three changes). Currently only 7 fields so cost is low.

### `ConnectedMessage.mode` Duplicates the `Mode` Union

`ws-protocol.ts:58` declares `mode: "default" | "plan" | "execute"` inline instead of importing `Mode` from `@diligent/protocol`. The `Mode` type is already exported from protocol. If a fourth mode is added to the protocol, `ws-protocol.ts` will silently fall out of sync — TypeScript won't catch the mismatch because the ws-protocol type is a subset.

**Impact**: Low now. One line to fix: `mode: Mode` from `@diligent/protocol`.

### `config/set` Intercepted Outside Protocol

`rpc-bridge.ts:148-176` intercepts `config/set` before forwarding to `appServer.handleRequest`. The client uses `requestRaw()` (an escape hatch that bypasses typed dispatch) to call it. This means model switching has no `@diligent/protocol` type coverage — it's a bespoke side channel.

This is a pragmatic workaround for a missing RPC method in the protocol, but it creates a gap: `config/set` will need formalization eventually. The current state (bridge intercept + untyped client call) should not be extended with additional out-of-protocol methods.

**Impact**: Low now, blocks protocol completeness. Document as a known gap in BACKLOG.

### `close()` Bug: Wrong Fallback Type for `userInput/request`

`rpc-bridge.ts:113-117`:
```typescript
pending.resolve({
  method: DILIGENT_SERVER_REQUEST_METHODS.APPROVAL_REQUEST,
  result: { decision: "reject" },
});
```
When a WebSocket session closes, all pending server requests are resolved with an approval-reject payload — including any pending `userInput/request`. The `toSafeFallback()` helper at line 36-48 correctly handles both request types, but `close()` bypasses it and hardcodes the approval variant. If a `userInput/request` is pending when the client disconnects, the agent receives a structurally invalid response.

**Fix**: Replace the hardcoded object in `close()` with `pending.resolve(toSafeFallback(request))`, where `request` is stored alongside the resolve function in `pendingServerRequests`.

**Impact**: Low frequency (requires pending user-input at exact disconnect moment), but a silent correctness bug.

## Forward Compatibility

### WS Reconnect Creates New Thread Instead of Resuming

When the WebSocket reconnects, `App.tsx:100-113` fires `onConnected` again and calls `thread/start` unconditionally, creating a brand-new thread. The previous thread (potentially mid-conversation) is orphaned from the session.

This behavior means:
- Any network hiccup (laptop sleep, wifi drop) loses the user's thread
- Thread history stays in the server DB but the client has no automatic recovery path
- The reconnect UI ("Retrying... 1/5") implies recovery is happening, but a new empty thread is actually being created

The fix is: store `state.activeThreadId` before disconnect; on reconnect, call `thread/resume` with the stored ID and fall back to `thread/start` only if it returns `found: false`.

**Horizon**: Already a user-facing regression today. Will become more visible as users run longer sessions.

### `tools.ts` Duplicates CLI Tool Assembly

`packages/web/src/server/tools.ts` is functionally identical to the CLI's tool assembly. Any new tool added to the CLI must be manually added to the web package too. Currently 12 tools + knowledge + task + collab. When MCP tools arrive in Phase 5, both files must be updated.

The fix is a shared `buildDefaultTools()` function in `@diligent/core`, exported from the tools layer, eliminating the duplication.

**Horizon**: Breaks in Phase 5 when MCP tools require synchronized updates across CLI and web.

### `ProviderName` Union Hardcoded in Web

`provider-manager.ts:13`: `type ProviderName = "anthropic" | "openai" | "gemini"`. This is a third copy of the provider union (after `config-writer.ts` and `provider.ts` in CLI). Adding a fourth provider requires updating this file too — and it's a different package, so it's easy to miss.

**Horizon**: Repeats the same class of bug as the `wizardEnterApiKey` miss from the previous assessment period (Action #1 in 2026-02-28-aac82c7).

### No Explicit Multi-Thread-Per-Session Support

`RpcBridge.sessions` maps one WebSocket to one `currentThreadId`. Switching threads (sidebar click → `openThread`) updates `session.currentThreadId` and orphans the previous thread from the threadOwners map:

```typescript
// rpc-bridge.ts:194-197
if (resumed.found && resumed.threadId) {
  session.currentThreadId = resumed.threadId;
  this.threadOwners.set(resumed.threadId, session.id);
  // ← previous thread still in threadOwners, pointing to this session
```

The previous thread remains in `threadOwners` mapping to the session — so notifications from the previous thread still route here. This is probably fine (user is still in the same session), but it means notifications from threads the user navigated away from will update the wrong (non-visible) state. This needs a cleanup step: remove the previous `threadId` from `threadOwners` when switching.

**Horizon**: Visible now as a subtle state corruption when switching between threads mid-turn.

## Priority Actions

### 1. Fix WebSocket Reconnect to Resume Previous Thread (UX Regression)
**What**: In `App.tsx:onConnected`, check `state.activeThreadId`. If present, call `thread/resume` with that ID; only fall back to `thread/start` if `found: false`.
**Why**: Users lose ongoing conversations on any network hiccup. The reconnect UI implies recovery.
**File**: `packages/web/src/client/App.tsx:100-113`

### 2. Fix `close()` to Use `toSafeFallback()` (Correctness Bug)
**What**: In `rpc-bridge.ts:close()`, store the original `request` in `pendingServerRequests` alongside `resolve`, then call `pending.resolve(toSafeFallback(pending.request))` on close.
**Why**: Wrong fallback type sent to agent when `userInput/request` is pending at disconnect.
**File**: `packages/web/src/server/rpc-bridge.ts:113-117`

### 3. Fix `threadOwners` Stale Entry on Thread Switch (State Corruption)
**What**: In the `thread/resume` handling block of `rpc-bridge.ts:193-198`, remove the previous `session.currentThreadId` from `threadOwners` before updating it to the new thread.
**Why**: Notifications from previous (non-active) threads continue routing to the session and updating invisible state.
**File**: `packages/web/src/server/rpc-bridge.ts:193-198`

### 4. Import `Mode` from `@diligent/protocol` in `ws-protocol.ts` (Type Sync)
**What**: Replace the inline `"default" | "plan" | "execute"` literal in `ConnectedMessage` with `Mode` imported from `@diligent/protocol`.
**Why**: One-line fix now; silent breakage later if a new mode is added.
**File**: `packages/web/src/shared/ws-protocol.ts:58`

### 5. Add `app-config.ts` Happy-Path Test
**What**: Add one test to `test/server.integration.test.ts` or a new `app-config.test.ts` that calls `loadWebRuntimeConfig` with a minimal fixture and asserts `model`, `streamFunction`, and `systemPrompt` are set.
**Why**: The most complex init path in the web package has zero test coverage. Future changes to knowledge/skill loading will have no regression guard.

### 6. Extract Shared `buildDefaultTools()` to Core (DRY — Pre-Phase 5)
**What**: Create `packages/core/src/tools/defaults.ts` exporting `buildDefaultTools(cwd, paths?, taskDeps?, collabDeps?)`. Replace both `packages/web/src/server/tools.ts` and the CLI equivalent with imports.
**Why**: Phase 5 MCP tool additions must hit both files. Extract now, before the count grows.
**Decision ref**: Prevents the same class of drift as Action #1 from 2026-02-28 assessment.
