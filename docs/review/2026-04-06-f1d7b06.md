## Forward Compatibility

### `ProviderBuiltinToolDefinition.capability` is extensible ‚Äî but `name` enum on `ProviderToolUseBlock` is not
**Horizon**: Breaks when a provider adds non-web native tools (e.g., code execution, file generation)

The `capability: "web"` field on `ProviderBuiltinToolDefinition` is designed as an enum ‚Äî adding `"code_execution"` would be a non-breaking extension. However, `ProviderToolUseBlock.name` is hardcoded to `z.enum(["web_search", "web_fetch"])` (data-model.ts:88). When a new provider-native capability arrives, this enum must be extended, which changes the Zod schema union discriminant. All content block consumers (web, cli, debug-viewer) that pattern-match on `name` will need updating.

The cheaper fix now: consider whether `name` should be `z.string()` with runtime validation rather than a compile-time enum, or keep it as an enum but document the extension protocol in P061.

**Evidence**: `packages/protocol/src/data-model.ts:88`, `packages/core/src/llm/types.ts:103`

### Plugin hook caching: unchanged, still per-turn reimport
**Horizon**: Same ‚Äî breaks when plugin count grows

`collectPluginHooks()` (plugin-loader.ts:57-92) still reimports all plugins on every UserPromptSubmit and Stop event. The studiorpc write-lock pattern (write-lock.ts:1-21) demonstrates that plugins need stable per-session state ‚Äî the write-lock works because `createTools()` runs once, not per-turn. If a plugin ever needs hook-scoped state (not just tool-scoped), the per-turn reimport would reset it.

**Evidence**: `packages/runtime/src/tools/plugin-loader.ts:57-92`, `thirdparty/overdare/plugins/plugin-studiorpc/src/write-lock.ts`

### TUI has no rendering for provider-native web tool content blocks
**Horizon**: Breaks user experience when TUI users encounter web search/fetch results

The web client has `AssistantContentBlocks.tsx` dispatching on `provider_tool_use`, `web_search_result`, and `web_fetch_result`. The CLI TUI has no equivalent ‚Äî these blocks will fall through to raw text rendering. Per CLAUDE.md: "When adding user-facing features, implement for both Web and TUI." This gap should be closed in the next period.

**Evidence**: absence of `provider_tool_use` handling in `packages/cli/src/tui/`

## Novel Perspective

### The Two Mega-Commits Reveal a Structural Testing Gap: Feature Branches Without Intermediate Verification

**Why no one has seen this**: Previous reviews focused on commit count, rework ratio, and remediation rate as velocity metrics. Those metrics treat commits as equal-weight units. This period's two mega-commits (4ba0be3: 46 files, 2,978 insertions; aa2df43: 68 files, 2,047 insertions) expose a different risk that unit-count metrics miss: **feature surface area without intermediate integration checkpoints**.

**Evidence**:

1. **Commit 4ba0be3 touches 46 files across 4 packages** in a single commit. It adds Zod schemas in protocol, type definitions in core, provider adapter logic in anthropic.ts and openai-shared.ts, tool config in runtime, and React components in web ‚Äî all atomically. If any layer had a subtle semantic mismatch (e.g., Anthropic's `server_tool_use` block having a field the normalized `ProviderToolUseBlock` drops), the only way to discover it is running the full system. There are no intermediate commits that could have been separately verified.

2. **Commit aa2df43 touches 68 files across 3 packages.** It adds `NativeCompactFn` contract in core, decision logic in compaction.ts, loop integration in loop.ts, session plumbing in runtime, and notification handling in thread-handlers.ts. The `compactionSummary` in-place mutation pattern (loop.ts:170) was introduced alongside the compaction decision logic, notification wiring, and UI indicator ‚Äî all in one shot.

3. **Both commits landed without regressions** ‚Äî the `f7c0ba0` rename and `219f222` stabilization that followed were minor. This is evidence that the author's mental model is accurate, but it's also survivorship bias. The risk is not that these particular commits broke something ‚Äî it's that this commit shape (multi-package atomic drops) scales poorly. At 50 files per commit, code review is difficult. At 100 files, it becomes impossible.

4. **The test suite caught the right things**: The `0bca4ca` fix (compaction summary notification) was a genuine bug found in the same period, and the `219f222` apply-patch stabilization addressed a real multi-hunk ordering issue. Both were caught and fixed quickly. But these were surface-level issues ‚Äî the deeper structural risks (like the `ThreadCompactStartResponseSchema` missing `summary`) were not caught by tests.

5. **Contrast with the 2026-04-05 remediation session**: That period's 4 PRs averaged 90 net LOC each and each addressed exactly one concern. The PRs were individually reviewable, separately revertible, and each had clear test coverage. This period's feature commits are none of those things ‚Äî they are correct, but they are not decomposed.

**Implication**: For future multi-package features, consider breaking implementation into per-package commits even within a single branch: protocol types first (verifiable with typecheck), then core adapters (verifiable with unit tests), then runtime integration (verifiable with integration tests), then frontend rendering (verifiable with component tests). This doesn't slow velocity ‚Äî the code is the same ‚Äî but it creates intermediate verification checkpoints and makes the git history reviewable. The P061 plan already had this structure (Task 1: contracts, Task 2: provider adapter, etc.) ‚Äî the implementation just landed as one commit instead of following the plan's task decomposition.
