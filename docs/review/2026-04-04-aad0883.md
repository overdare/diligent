## Novel Perspective

### The Remediation Pipeline Has a Structural Frequency Problem, Not a Priority Problem

**Why no one has seen this**: Previous reviews attributed 0% remediation periods to "development focus" shifting elsewhere — analytics features, product work, planning sessions. This frames remediation as a priority decision: the developer chose product work over tech debt. But the data across 5 review periods reveals a structural pattern that priority framing obscures.

**Evidence**:

1. **The alternating pattern is too regular to be coincidence**: Over the last 5 periods, remediation rates are 75%, 25%, 0%, 75%, 0%. High-remediation periods alternate with low/zero periods. This is not a priority call — it's a workflow artifact. The solo developer appears to have two distinct work modes: "code sessions" where PRs land and remediation happens naturally, and "product/planning sessions" where no source code changes at all. Remediation only happens as a byproduct of code sessions.

2. **Priority Actions are never worked independently**: Every resolved Priority Action in the last 5 reviews was resolved as part of a PR that also delivered feature or fix work. PR #73 (stop_hook_active) landed alongside #75 (resetTurnRuntimeState), #77 (JSONC parser), and #78 (data extraction helpers) — all in the same code session. No Priority Action has ever been resolved in isolation as a standalone remediation commit. This means remediation throughput is coupled to feature delivery cadence, not to action urgency.

3. **The hook dispatch extraction (#2) has survived 4 reviews not because it's deprioritized, but because it requires a dedicated code session**: The extraction involves modifying 3 files (server.ts, thread-handlers.ts, runner.ts) and adjusting tests. It's a focused 30-minute task, but it doesn't naturally attach to any feature PR. In a workflow where remediation only happens alongside feature work, standalone refactors have no vehicle to land.

4. **Planning sessions produce no remediation by definition**: This period's 3 commits are all documentation and build tooling. The P060 plan (515 LOC of markdown) represents significant intellectual work, but it generates zero remediation. If the developer's next session is also planning-focused (e.g., implementing P060's design), Priority Actions will survive to a 5th review.

5. **The e2e test suite has grown to 8 files covering core protocol flows**: `packages/e2e/` now covers conversation, custom agents, mode/config, multi-connection, protocol lifecycle, session resume, turn execution, and WebSocket transport. This is a solid foundation that would support the refactoring work in Priority Actions — but the tests aren't being used as refactoring enablers because refactoring isn't happening in dedicated sessions.

**Implication**: Consider batching Priority Actions into the first 30 minutes of each code session as a "warm-up" practice, regardless of the session's main goal. The hook dispatch extraction, PluginHookInput expansion, and render-payload split are all 15-30 minute tasks. They don't need a dedicated session — they need a habit of starting each code session with one remediation item before switching to the main feature work. This breaks the structural coupling between remediation and feature delivery.
