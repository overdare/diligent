## Cross-Package Coherence

### 9. Multi-Client Subscription System — New Cross-Package Feature
**Scope**: packages/core, packages/web, packages/protocol
**Impact**: Significant new infrastructure, well-designed

The P029 multi-client thread subscription system (942386c, e037f54) adds:

- **Protocol**: `DILIGENT_WEB_REQUEST_METHODS.THREAD_SUBSCRIBE/UNSUBSCRIBE` in protocol methods
- **Server**: `RpcBridge` manages `threadSubscribers` map, per-thread routing, and pending request re-delivery on reconnect
- **Client**: `rpc-client.ts` subscribe/unsubscribe methods

Key design properties:
- Notifications route only to subscribers of the relevant thread (not broadcast)
- Server requests (approval, user input) fan out to all thread subscribers, first responder wins
- Disconnect cleanup: subscriptions removed on WebSocket close, pending requests survive for re-delivery
- Turn initiator tracking: `turnInitiators` map prevents echo of `userMessage` items to the sender

This is a well-designed peer model — no "owner" concept, any subscriber can respond to requests. The one weakness is the `as` casts for subscribe/unsubscribe params (`rpc-bridge.ts:287, 308`) — these should have Zod schemas like the other web methods.

### 10. Collab Event System — Sub-Agent Visibility
**Scope**: packages/core, packages/web, packages/protocol
**Impact**: Major new feature, clean layer separation

The collab event system (a6f9860, 1f00b53) adds real-time visibility into sub-agent execution:

- **Core**: `AgentRegistry` emits `CollabAgentEvent` via callback → `SessionManager` bridges to `ProtocolNotificationAdapter` → server notifications
- **Protocol**: 9 new collab notification methods (`collab/spawn/begin`, `collab/spawn/end`, etc.) in `DILIGENT_SERVER_NOTIFICATION_METHODS`
- **Web**: `thread-store.ts` reduces collab events into `RenderItem` with `kind: "collab"`, `CollabEventBlock.tsx` and `CollabGroup.tsx` render them

The event type hierarchy is clean:
```
AgentRegistry.emit(CollabAgentEvent) → SessionManager → ProtocolNotificationAdapter → DiligentAppServer → RpcBridge → WebSocket → thread-store reducer → CollabEventBlock
```

Each layer transforms only its own concern. Adding a new collab event type requires: (1) add to `AgentEvent` union, (2) add notification schema in protocol, (3) add reducer case in thread-store, (4) add render component. This is a 4-location change but each is in a different layer — reasonable for a cross-cutting feature.

## Forward Compatibility

### 13. ModelClass System — New Extension Point
**Horizon**: Immediate use for sub-agent model selection

`packages/core/src/provider/models.ts` adds `ModelClass` ("pro" | "general" | "lite") with `resolveModelForClass()` and `agentTypeToModelClass()`. This allows sub-agents to use cheaper models for simple tasks:

```typescript
// AgentRegistry resolves model class before spawning
const targetClass: ModelClass = params.modelClass ?? agentTypeToModelClass(params.agentType, this.deps.model);
const childModel = resolveModelForClass(this.deps.model, targetClass);
```

The resolution logic falls through correctly: if a class isn't available, it uses the parent's model. This is a well-designed extension point that will support future model routing policies.

### 14. Provider Registry — Still If/Else Chain
**Horizon**: Adding 4th provider (likely DeepSeek or local models)

`packages/core/src/provider/provider-manager.ts:179-185` still uses an if/else chain:

```typescript
if (provider === "openai") {
  stream = createOpenAIStream(apiKey, this.baseUrls.openai);
} else if (provider === "gemini") {
  stream = createGeminiStream(apiKey, this.baseUrls.gemini);
} else {
  stream = createAnthropicStream(apiKey);
}
```

The else fallback silently defaults to Anthropic for unknown providers — the last silent failure point. Adding a 4th provider still requires edits in:
1. `protocol/src/data-model.ts` — `ProviderNameSchema` enum
2. `core/src/provider/provider-manager.ts` — `PROVIDER_NAMES`, `DEFAULT_MODELS`, `PROVIDER_HINTS`, if/else chain
3. `core/src/auth/auth-store.ts` — `AuthKeysSchema`
4. `core/src/provider/` — new stream factory file

TypeScript catches omissions at 2-3 via `Record<ProviderName, ...>` exhaustiveness. The if/else at step 2 is the only location that silently falls through.
