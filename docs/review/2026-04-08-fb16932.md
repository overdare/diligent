## Cross-Package Coherence

### Third transcript renderer emerges in VS Code webview
**Scope**: apps/vscode-extension, packages/web, packages/cli

The VS Code extension's `app.ts` (618 LOC) implements a third independent transcript rendering pipeline:

| Concern | Web | CLI | VS Code |
|---------|-----|-----|---------|
| Renderer | React components | Custom ANSI | Raw DOM string builder |
| Event reduction | `thread-store.ts` + `collab-reducer.ts` | `thread-event-reducer.ts` + `thread-store-utils.ts` | `applyAgentEvents()` inline in `app.ts:448-540` |
| ThreadItem rendering | `AssistantContentBlocks.tsx` | `transcript-render.ts` | `renderThreadItem()` in `app.ts:183-250` |
| Live streaming state | React state hooks | ThreadStore primitives | Manual `ConversationViewState` object |
| Total LOC | ~1,232 | ~1,774 | ~618 |

The VS Code renderer is the lightest because it skips collab event visualization and provider-native block rendering. But it still independently handles:
- AgentEvent → state reduction (15 event types in `applyAgentEvents`, lines 448-540)
- ThreadItem → HTML rendering (`renderThreadItem`, lines 183-250)
- Live streaming state management (text, thinking, tool progress, overlay status)
- Markdown rendering via `marked` (separate from web's React-markdown and CLI's marked+ANSI)

**Why this matters:** Every new protocol event type or rendering change now requires updates in three places. The VS Code extension does not yet render provider-native blocks (`provider_tool_use`, `web_search_result`), collab timeline events, or `request_user_input` tool results — these are all gaps that will need to be filled. Each gap filled adds to the maintenance surface.

**Current mitigation:** The extension correctly uses `ProtocolNotificationAdapter.toAgentEvents()` from protocol for notification → event mapping, so the event shape is consistent. The divergence is only in the rendering layer.

## Novel Perspective

### The Notification-to-Render Pipeline Needs a Shared Intermediate Representation

- **Why no one has seen this**: Previous reviews focused on the TUI-Web divergence as a rendering problem (different UI frameworks require different output). The VS Code extension reveals that the divergence is actually one layer deeper — there is no shared intermediate representation between "protocol notifications arrive" and "UI renders them." Each client independently implements notification → state reduction → item rendering, duplicating not just rendering code but also the state machine logic.

- **Evidence**:
  - Web: `thread-store.ts` (641 LOC) + `collab-reducer.ts` (283 LOC) = 924 LOC of state reduction
  - CLI: `thread-event-reducer.ts` (398 LOC) + `thread-store.ts` (576 LOC) = 974 LOC of state reduction
  - VS Code: `app.ts:applyAgentEvents()` (92 LOC) + `renderThreadItem()` (67 LOC) = 159 LOC of state reduction (stripped-down version)
  - All three implement the same conceptual state machine: "given a stream of AgentEvents, produce a conversation state with live text, live tool progress, completed items, and overlay status"
  - `ProtocolNotificationAdapter` (`packages/protocol/src/notification-adapter.ts`) already solves half the problem by normalizing notifications to `AgentEvent[]` — but no shared component takes `AgentEvent[]` and produces a renderer-agnostic conversation state

- **Implication**: A shared `ConversationStateReducer` in `@diligent/protocol` (or a new `@diligent/client-state` package) could own the `AgentEvent[] → ConversationState` reduction logic. Each client would then only implement `ConversationState → UI`, which is genuinely framework-specific. This would:
  1. Reduce the surface area for protocol event additions from 3 clients to 1 reducer + 3 renderers
  2. Make the VS Code extension automatically gain collab and provider-native block support when the reducer adds them
  3. Prevent the state machine bugs that arise from implementing the same 15-event-type switch statement independently
  4. Keep the rendering layer (React, ANSI, DOM strings) appropriately client-specific

  The cost is moderate: extracting the common state shape requires agreeing on a framework-agnostic `ConversationState` type and refactoring existing clients to consume it. The VS Code extension's `ConversationViewState` (`src/views/webview/protocol.ts:5-19`) is already close to this shape and could serve as the starting point.

## Persistent Issues

### TUI-Web-VSCode transcript rendering divergence
- **First raised**: 2026-03-13 (435614c) — originally as TUI-Web divergence
- **Appearances**: 20+ reviews (435614c through 2026-04-08)
- **Status trajectory**: Getting worse — expanded from 2 renderers to 3 with VS Code extension
- **Current evidence**: Web (~1,232 LOC), CLI (~1,774 LOC), VS Code (~618 LOC) for combined thread-store + rendering. Three independent `AgentEvent → UI` pipelines with no shared state reduction layer.
- **Why it persists**: Each client uses a different UI framework (React, custom ANSI, raw DOM) making full unification impractical. The missing piece is a shared intermediate state reducer (see Novel Perspective). The VS Code extension's lightweight approach validates that a common `ConversationState` shape is achievable — its `ConversationViewState` type is framework-agnostic.
