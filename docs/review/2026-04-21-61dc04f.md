**Impact**: Compounds over time

The configurable storage namespace (`DILIGENT_STORAGE_NAMESPACE` env var, `.diligent/` → `.overdare/` migration) is a cross-cutting change that touches:

## Cross-Package Coherence

### Dual storage namespace implementations create a new sync surface
**Scope**: runtime, overdare-ai-agent

The storage namespace logic now exists in two languages:

- **Rust**: `apps/overdare-ai-agent/src/storage.rs` (163 LOC) — directory resolution, namespace migration, platform-specific home directory
- **TypeScript**: `packages/runtime/src/infrastructure/diligent-dir.ts` (67 LOC) — directory resolution, gitignore setup

The Rust side owns migration (`.diligent/` → `.overdare/`), the TypeScript side owns directory creation and gitignore. This is a reasonable split IF the Rust launcher always runs before the TypeScript runtime. But if a user runs the Diligent CLI directly (not through overdare-ai-agent), migration never happens and the TS runtime falls back to `.diligent/` — which is correct by default.

The risk is not in the current state but in future evolution: any change to directory structure, new subdirectories, or namespace rules must be coordinated across both implementations with no shared type contract.

Evidence: `apps/overdare-ai-agent/src/storage.rs`, `packages/runtime/src/infrastructure/diligent-dir.ts`

## Forward Compatibility

### Storage namespace migration is Rust-only — TypeScript runtime has no migration path
**Horizon**: Breaks when generic Diligent users want to switch namespace, or when the TS runtime needs to handle legacy directories

`apps/overdare-ai-agent/src/storage.rs` implements `migrate_global_namespace_if_needed()` and `migrate_local_namespace_if_needed()` — renaming `.diligent/` to `.overdare/`. The TypeScript runtime in `packages/runtime/src/infrastructure/diligent-dir.ts` has no migration logic. It resolves the namespace from env var and creates directories, but never checks for or migrates legacy directories.

Currently this works because the Rust launcher always runs first. But if the product expands to scenarios where the TS runtime starts without the Rust launcher (e.g., direct `bun` invocation, web-only deployment, or a different product variant), legacy directories will be silently ignored.

Evidence: `apps/overdare-ai-agent/src/storage.rs:50-63`, `packages/runtime/src/infrastructure/diligent-dir.ts` (no migration functions)

### Path normalization logic duplicated across language boundaries
**Horizon**: Diverges when new path edge cases emerge

Windows path normalization (`/c/path` → `C:\path`) is implemented independently in:
- `apps/overdare-ai-agent/src/webserver.rs:18-48` — Rust, for cwd argument normalization
- `packages/core/src/llm/local-image-paths.ts:27-33` — TypeScript, for persisted image path resolution

Both solve overlapping but not identical problems. The Rust version handles `/Users/` → `SystemDrive:\Users\` mapping; the TypeScript version handles POSIX-style cwd detection. Neither references the other's logic. Future Windows edge cases (network paths, WSL interop, junction points) will need coordinated fixes in both.

## Novel Perspective

### The Rust launcher introduces a language boundary that the architecture doesn't account for — and it's already producing behavior divergence

- **Why no one has seen this**: Previous reviews evaluated `apps/` as thin wrappers (Tauri desktop, VS Code extension) that delegate all behavior to the TypeScript runtime. The new `overdare-ai-agent` Rust binary breaks this assumption: it makes autonomous decisions about storage layout, path normalization, namespace migration, and process lifecycle that the TypeScript runtime must implicitly agree with.

- **Evidence**:
  1. **Storage defaults diverge**: Rust defaults to `"overdare"` namespace (`storage.rs:4`), TypeScript defaults to `"diligent"` (`diligent-dir.ts:5`). Both are correct for their context, but there is no shared constant, schema, or test that verifies they agree when configured identically.
  2. **Migration is one-sided**: Only Rust performs `.diligent/` → `.overdare/` migration (`storage.rs:50-63`). TypeScript creates directories but never migrates (`diligent-dir.ts:51-66`). If the Rust launcher skips migration (e.g., error, race condition, first-run before init), the TypeScript runtime will create a fresh `.overdare/` directory alongside the existing `.diligent/` — silently splitting state.
  3. **Path normalization reimplemented**: `webserver.rs:18-48` and `local-image-paths.ts:27-33` both handle Windows path conversion independently. The Rust version has a special `/Users/` → `SystemDrive:\Users\` case that the TypeScript version doesn't have, and vice versa for POSIX-style cwd detection.
  4. **No cross-language contract**: There is no shared test, config schema, or integration check that verifies the Rust and TypeScript implementations produce compatible directory structures. The only implicit contract is the `DILIGENT_STORAGE_NAMESPACE` env var name.

- **Implication**: The project needs to decide whether the Rust launcher is a "dumb process supervisor" (just spawns Bun with correct args) or an "infrastructure peer" (makes its own decisions about storage and paths). If the latter — which the current code suggests — then:
  1. Add a design decision (D099) documenting the Rust/TypeScript boundary contract: what the Rust launcher owns, what it delegates, and the shared invariants
  2. Add an integration test that exercises the full launch path: Rust `init` → Rust `webserver` → TypeScript runtime startup, verifying that storage directories, namespace, and paths are consistent
  3. Consider whether migration logic should live in TypeScript runtime (where all other `.diligent/` logic lives) rather than in the Rust launcher, to prevent the one-sided migration gap
