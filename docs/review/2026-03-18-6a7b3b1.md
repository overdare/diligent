### 2. ProtocolNotificationAdapter Silent Fork — New Cross-Package Duplication
**Impact**: Compounds with every new notification type or field addition

The P054 commit (feabee1) also created a **second ProtocolNotificationAdapter** in the web package:

| Location | LOC | Import source | Consumer |
|----------|-----|---------------|----------|
| `packages/runtime/src/notification-adapter.ts` | 319 | `@diligent/core`, `./agent-event` | CLI (via `@diligent/runtime`) |
| `packages/web/src/client/lib/protocol-notification-adapter.ts` | 345 | `@diligent/protocol` only | Web (browser-safe) |

Both classes have the same name, the same `toAgentEvents()` method signature, and identical switch-case structures. But they have already diverged:

1. **Web adds timestamp fields**: `handleItemStarted` passes `item.timestamp` and `item.startedAt` to events. Runtime does not.
2. **Web adds duration fields**: `handleItemCompleted` passes `item.timestamp`, `item.reasoningDurationMs`, `item.turnDurationMs`, `item.durationMs`. Runtime does not.
3. **Web adds helper exports**: `findModelInfo`, `supportsThinkingNone`, `getThinkingEffortOptions`, `getThinkingEffortUsage` (40 LOC of model utility functions bundled into the adapter file).

The runtime adapter's `@summary` still says "Shared protocol notification → AgentEvent adapter for TUI and **Web** frontends" — but web no longer imports it. This is a textbook silent fork: the code was copied for browser-safety reasons, immediately gained web-specific fields, and will diverge further with every protocol change.

The divergence mechanism is identical to the resolved tool-info.ts problem: each consumer independently adds fields it needs, and no compile-time or test-time mechanism detects when they fall out of sync. Adding a new notification type (e.g., `THREAD_ARCHIVED`) requires updating both adapters independently.

## Novel Perspective

### The Notification Adapter Fork Reveals a Systemic Browser-Safety Boundary Problem

- **Why no one has seen this**: Previous reviews tracked duplication as a type-sync or naming-consistency problem (ProviderName, Mode/ModeKind). The new ProtocolNotificationAdapter fork was categorized as a P054 implementation detail, not a systemic pattern. But it exposes a deeper structural tension: **the web client cannot import from `@diligent/runtime` in browser code**, so any logic shared between CLI (which uses runtime) and web (which uses protocol-only in browser) must be either (a) duplicated, (b) moved to protocol, or (c) restructured to live in a browser-safe subset.

- **Evidence**:
  1. **ProtocolNotificationAdapter**: Runtime version imports `AgentEvent` from `./agent-event` (runtime-internal type). Web version imports `AgentEvent` from `@diligent/protocol` (protocol type). These are **different types with the same name** — runtime's `AgentEvent` extends core's `CoreAgentEvent` with runtime-specific events (`agent-event.ts:1-33`), while protocol's `AgentEvent` is a Zod-inferred type. The web adapter can never import the runtime adapter because it would pull in Node.js-only code.

  2. **notification-adapter.ts @summary is now incorrect**: It says "Shared protocol notification → AgentEvent adapter for TUI and Web frontends" — but web no longer uses it. The CLI uses it via `@diligent/runtime`. This stale documentation masks the fork.

  3. **The pattern will recur**: Any future shared logic between CLI and web that requires `AgentEvent` processing faces the same choice: duplicate for browser safety, or refactor the type hierarchy. Examples: shared event filtering, shared event aggregation for usage tracking, shared event replay for testing.

  4. **The `AgentEvent` type itself is the root cause**: Protocol defines `AgentEvent` as a broad union type (Zod schema). Core defines `CoreAgentEvent` as a narrower tagged union. Runtime extends it with `CollabEvent` variants. Web needs the protocol version (browser-safe). CLI needs the runtime version (richer events). The same event concept has 3 type definitions across 3 packages, each with different members.

  5. **Quantified impact**: 664 LOC (319 + 345) of notification adapter logic, ~280 LOC identical, ~80 LOC divergent, ~40 LOC web-only helpers. Each new notification case adds ~10-15 LOC to each adapter independently.

- **Implication**: The browser-safety boundary creates a "fork pressure" on any shared protocol-processing logic. The notification adapter is the first victim, but transcript parsing utilities, event filtering, and usage aggregation face the same pressure. The structural fix is to ensure all protocol-processing logic lives in `@diligent/protocol` (browser-safe by definition) or in a new browser-safe re-export from runtime. Specifically:
  - Move `ProtocolNotificationAdapter` to `@diligent/protocol` (it already uses only protocol types in the web version)
  - Have runtime re-export it, and CLI/web both import from their respective packages
  - The runtime version's `@summary` should be updated immediately regardless, since it's misleading

  This prevents the fork from compounding. The cost is moving one file and updating imports. The payoff is eliminating the highest-risk new duplication in the codebase.
