## Forward Compatibility

### handleTurnStart() is the next growth attractor
**Horizon**: Breaks when next turn-level capability arrives (e.g., mid-turn model switching, parallel tool execution)

`thread-handlers.ts:738` LOC is stable but `handleTurnStart()` is approximately 370 LOC combining: turn initialization, hook execution, message preparation, agent loop setup, and error handling. This is the largest single function in the runtime. The next turn-level feature will compound its complexity.

Natural decomposition: normalize → hooks → execute (three focused functions). This is not urgent but should be done before the next turn-capability addition.

Evidence: `packages/runtime/src/app-server/thread-handlers.ts`

### AgentEvent schemas in data-model.ts have circular-dependency constraint
**Horizon**: Blocks further data-model decomposition

PR #112 documents that `AgentEvent` schemas (~160 LOC) remain in `data-model.ts` because extracting to `events.ts` would create circular imports: `events.ts` would need `AssistantMessage`/`UserMessage` from `data-model.ts`, while `data-model.ts` re-exports from `events.ts`. This is a known constraint that limits further decomposition without restructuring the schema dependency graph.

This is acceptable at 440 LOC but should be revisited if `data-model.ts` grows past ~500 LOC again.

### Collab registry at 654 LOC — growing orchestration surface
**Horizon**: Breaks when nested agent features expand (thread fork + sub-agent resume across sessions)

`packages/runtime/src/collab/registry.ts` (654 LOC) manages the full sub-agent lifecycle: spawn, wait, join, steering, shutdown, resume. It's the third-largest runtime file and handles complex state (agent sessions, reference tracking, abort coordination). Future features like thread fork with sub-agent state or cross-session agent resume will compound its complexity.

Evidence: `packages/runtime/src/collab/registry.ts:654`

- **Implication**: Priority Actions in this review should be optimized for automation-friendliness. This means:
  1. Each action should be self-contained with clear file scope (the bot does well with bounded extractions and delegations)
  2. Actions requiring design judgment or cross-cutting behavioral changes should be flagged as manual-only

  The risk is that reviews start generating actions optimized for what the bot CAN do (extractions, splits, mechanical adoptions) rather than what the project NEEDS (design decisions, capability planning, hard trade-offs). This review should be the first to explicitly distinguish between automation-suitable and judgment-requiring actions.
