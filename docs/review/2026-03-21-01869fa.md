## Structural Integrity

**Impact**: Cosmetic

The runtime version is not exported and used only in config validation, so divergence risk is low. However, it is a sync point that could silently break if protocol updates the enum.

### D089 (Thread Fork) and D091 (Thread Archive) — Decided but Not Implemented
**Impact**: Compounds over time

Both decisions exist in `docs/plan/decisions.md` but have no corresponding implementation in the session format (SESSION_VERSION remains at 8). No `forkedFromId` field in `SessionHeader`, no `ArchiveEntry` in `SessionEntry` union, no `thread/fork` or `thread/archive` methods in `packages/protocol/src/methods.ts`. As more features build on the current session format, retrofitting these will become increasingly expensive — every consumer of session entries needs updating.

### Protocol Version Hardcoded
**Impact**: Compounds over time

`packages/runtime/src/app-server/server.ts:341` returns `protocolVersion: 1` with no negotiation mechanism. Before any breaking protocol change (e.g., D089/D091 features), a version handshake will be required. The longer this is deferred, the more clients assume protocol v1 is the only version.

### Notification Adapter Consolidation — Complete
**Impact**: Resolved

Previous reviews flagged the ProtocolNotificationAdapter fork between runtime (319 LOC) and web (345 LOC). This is now resolved: `packages/runtime/src/notification-adapter.ts` is 3 LOC (re-export from protocol), and the canonical implementation lives in `packages/protocol/src/notification-adapter.ts` (314 LOC). Clean single source of truth.

### Type Flow: Excellent
**Scope**: All packages

The `protocol → core → runtime → cli/web` type chain is clean:

- `Mode`: single definition in protocol, consumed everywhere
- `ModelInfo`: properly dual-defined (Zod in protocol for wire, interface in core for runtime)
- Zero `any` usage across the entire codebase
- All web/cli imports verified correct (`@diligent/protocol` for RPC types, `@diligent/core` for runtime types)

### Transport Parity: Excellent
**Scope**: CLI, Web, Desktop

All three transports delegate to the single `DiligentAppServer` class. Transport adapters (stdio for CLI, WebSocket for web, Tauri sidecar for desktop) are thin (~150 LOC each). Session lifecycle, auth flows, and provider management use identical handler functions via `dispatchClientRequest()`. No behavioral divergence detected.

### Plugin-studiorpc Type Independence: Intentional and Clean
**Scope**: thirdparty/overdare/plugins/plugin-studiorpc

The studiorpc plugin declares its own `Tool`, `ToolContext`, `ToolResult`, and `ToolRenderPayload` interfaces (`src/tool-types.ts`, 54 LOC) rather than importing from `@diligent/core`. This is correct per P032's full-trust plugin contract — plugins are npm packages that export `createTools()`, not direct consumers of core internals. The plugin's `ToolContext.approve()` pattern (approval gates with read/write/execute permissions) extends beyond core's tool interface without creating coupling.

### Thread-Store State Management Divergence
**Scope**: packages/web vs packages/cli

Web's `thread-store.ts` (597 LOC) uses immutable reducer functions. CLI's `thread-store.ts` (804 LOC) uses a mutable class with side effects (timers, cache mutations). Both correctly import shared types from `@diligent/protocol`, but any new event type or collab feature requires parallel implementation in two incompatible paradigms. This was flagged in the 2026-03-20 review's Novel Perspective — the divergence persists.

## Forward Compatibility

### Plugin API Version Lock
**Horizon**: Breaks when plugin contract needs breaking changes (e.g., adding `logger` or `config` to createTools context)

`packages/runtime/src/tools/plugin-loader.ts:138-141` hardcodes `apiVersion` to `1.x` with no migration path. The `createTools()` context is minimal (`{ cwd }`) — any expansion (logger, config, capabilities) breaks the contract. The current approach has no deprecation/migration mechanism. Fix now: add optional context fields (non-breaking) or define a version negotiation protocol. Fix later: requires coordinating breaking changes across all deployed plugins.

### Pending Steering Messages Not Persisted
**Horizon**: Breaks when session recovery becomes critical (e.g., long-running agent tasks, desktop app restart)

`packages/runtime/src/session/manager.ts:53` holds `pendingMessages` as an in-memory array. If the process restarts between message queuing and turn execution, steering messages are lost. Currently low-impact (sessions are short-lived), but becomes a real problem for long-running collaboration sessions or desktop hibernation scenarios.

### Thread-Scoped Method List is Static
**Horizon**: Breaks when adding plugin-contributed thread-scoped methods

`packages/runtime/src/app-server/server.ts:307-319` defines `THREAD_SCOPED` as a static string array. New thread-scoped protocol methods require code changes here. If plugins can contribute thread-scoped methods (not yet possible but a natural extension of P032), this static list becomes a bottleneck. Consider deriving from method metadata or the protocol definition.

## Novel Perspective

### The Thirdparty Plugin is a Shadow Integration Test Suite

**Why no one has seen this**: Reviews have consistently focused on the internal package architecture (protocol→core→runtime→cli/web), treating thirdparty/ as an external concern outside the review scope. But the studiorpc plugin's 6-commit development arc this period is actually the first real-world validation of the plugin contract (P032) — and it reveals contract gaps that unit tests and e2e tests don't cover.

**Evidence**:
- `thirdparty/overdare/plugins/plugin-studiorpc/src/tool-types.ts` (54 LOC) — the plugin had to re-declare `Tool`, `ToolContext`, `ToolResult`, and `ToolRenderPayload` from scratch. These types structurally overlap with `@diligent/core`'s tool types but cannot import them because P032's contract is `createTools()` with minimal context. The plugin author (same developer) knows the core types intimately, so the redeclaration is faithful — but a third-party author would have no such guarantee.
- `plugin-loader.ts:175` wraps plugin tools with a legacy `approve`/`ask` context adapter — the plugin contract's `ToolContext` shape differs from core's, requiring a compatibility shim at runtime.
- The studiorpc plugin invented its own approval model (`permission: "read" | "write" | "execute"`) because the plugin contract provides no structured approval API. This works for this plugin but means every plugin will invent its own approval semantics.
- `instance-upsert-tool.ts:223` — the plugin manages file I/O, RPC calls, and approval flows entirely independently, duplicating patterns that exist in runtime's built-in tools but aren't exposed as composable utilities.

**Implication**: The plugin contract (P032) was designed as a "trust boundary" — minimal surface, no core coupling. This is architecturally sound. But the studiorpc development reveals that the contract is too minimal: plugins need structured approval, render payload types, and composable I/O patterns. Rather than waiting for a third-party developer to hit these walls, the studiorpc experience should be documented as a P032 retrospective and used to inform the plugin contract's evolution (e.g., optional typed context fields, shared utility exports). The plugin SDK doesn't need to be `@diligent/core` — it could be a thin `@diligent/plugin-sdk` package exporting just the types and utilities that plugin authors need.
