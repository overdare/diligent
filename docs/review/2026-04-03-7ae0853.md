## Forward Compatibility

### render-payload.ts is the new growth attractor
**Horizon**: Compounds each time a tool gains custom rendering

render-payload.ts grew from 903 → 995 LOC this period through new `RenderStrategy` map entries. The strategy pattern is correct — each tool's rendering is isolated in its own entry. But the file now holds all tool render strategies, all generic render helpers, all patch parsing, and all search/knowledge/plan render input types. Unlike server.ts (which was flagged at 1,000 LOC), render-payload.ts is lower-risk because its functions are pure and stateless. However, the file is approaching the point where navigating between strategies becomes friction.

**When this breaks**: When plugins need custom render strategies. Currently the `RENDER_STRATEGIES` map is a hardcoded `Map<string, RenderStrategy>`. Plugin-authored tools (like studiorpc methods) cannot register their own render strategies — they must build `ToolRenderPayload` directly in the plugin. If the render strategy pattern is valuable (it is), plugins should be able to participate.

**Evidence**: `packages/runtime/src/tools/render-payload.ts` (995 LOC), `RENDER_STRATEGIES` map at line 40

### Hook dispatch duplication remains — but risk reduced
**Horizon**: Same as prior 3 reviews — compounds when new hook events are added

The shell-then-plugin dispatch pattern still exists independently in `server.ts:916-943` (Stop hooks) and `thread-handlers.ts:382-409` (UserPromptSubmit hooks). However, the compounding risk factors from previous reviews have been mitigated:
- Data extraction helpers are now in `input-builder.ts` (PR #78) — reducing server.ts coupling

The extraction is still valuable for preventing copy-paste when a third hook event is added, but it's no longer the most urgent structural concern.

**Evidence**: `packages/runtime/src/app-server/server.ts:913-950`, `packages/runtime/src/app-server/thread-handlers.ts:382-434`

## Novel Perspective

### The Overdare Plugin Ecosystem Is Developing Its Own Architectural Gravity

**Why no one has seen this**: Previous reviews tracked Overdare plugins as third-party content — appropriately scoped under `thirdparty/` and isolated from core package analysis. But this period, 55% of commits were Overdare-scoped. The plugin-studiorpc plugin now has 29 source files with its own render system, tool registry, method modules, and test suite. Combined with plugin-analytics, plugin-rag, plugin-validator, and dedicated skills under `thirdparty/overdare/skills/`, the Overdare integration has become a de facto product vertical — larger in surface area than several core packages.

**Evidence**:

1. **Commit gravity**: 16/29 commits this period touched Overdare. The prior period (db352b7) had 5/10 analytics-related commits. Two consecutive reviews where Overdare work exceeds 50% of total commits.

2. **Cross-boundary feature pressure**: Commit 12d41af ("simplify Studio RPC error messages") added 96 lines to `packages/runtime/src/tools/render-payload.ts` — a core runtime file. The RPC error simplification logic serves the Overdare Studio connection specifically (local-only connection diagnostics). This is the first time an Overdare-specific UX concern has driven core runtime changes. If more Studio RPC-specific render strategies land in render-payload.ts, the core runtime begins to accumulate product-specific rendering logic.

3. **The plugin has its own render layer**: `plugin-studiorpc/src/render.ts` (360 LOC) builds tool render payloads independently from the runtime's `RENDER_STRATEGIES` map. This is correct today — plugins build `ToolRenderPayload` directly. But the plugin's render.ts is now larger than most core package files, and it duplicates summary/truncation patterns found in render-payload.ts (e.g., `summarizeText()` vs runtime's `summarizeRenderText()`).

4. **Workspace configuration includes thirdparty plugins**: `package.json:10` includes `"thirdparty/*/plugins/*"` as workspace members. These plugins participate in Bun workspace resolution, meaning they can import from `@diligent/*` packages but aren't subject to the same CI typecheck pipeline (the `typecheck` script explicitly lists only core packages). A type error in a thirdparty plugin won't fail CI.

5. **Skills add another surface**: `thirdparty/overdare/skills/` contains product-specific skills (e.g., `tpa/SKILL.md`) that are loaded by the runtime's skill discovery system. These are injected into the system prompt alongside core skills. A misconfigured Overdare skill could affect prompt token budget for all sessions.

**Implication**: Consider whether Overdare-specific render logic in `render-payload.ts` should migrate to plugin-studiorpc's own render module (using `ToolRenderPayload` directly, as the plugin already does). More broadly, evaluate whether the `thirdparty/*/plugins/*` workspace pattern needs a lightweight CI gate — at minimum running `tsc --noEmit` on thirdparty plugins — to catch type drift between plugin code and the SDK it targets. The growing Overdare surface area is not a problem per se, but the blurred boundary between "core platform" and "first-party product integration" creates risk: core runtime changes that serve Overdare specifically may not be the right abstraction for future non-Overdare plugins.
