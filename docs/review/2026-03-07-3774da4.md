## Structural Integrity

### 1. Unified Transport Architecture (P036–P039) — Architectural Milestone
**Impact**: High — eliminates the deepest structural complexity in the codebase

Commit e5a29b9 (70 files, +5,872/-2,221) is the single most impactful commit in the project's history. It consolidates all protocol handling into `DiligentAppServer`:

| Before | After |
|--------|-------|
| `web/src/server/rpc-bridge.ts` (864 LOC) — request routing, session state, `as` casts, multi-client subscriptions, auth flows | **Deleted** |
| `web/src/server/index.ts` (~160 LOC) — Express server + rpc-bridge wiring | `web/src/server/index.ts` (277 LOC) — pure WebSocket transport |
| `web/src/shared/ws-protocol.ts` (~90 LOC) — redundant types + re-exports | `ws-protocol.ts` (4 LOC) — single `ModelInfo` re-export |
| `core/src/app-server/server.ts` (~810 LOC) — core protocol handling | `server.ts` (1,437 LOC) — **universal** protocol handling |
| `protocol/src/web-requests.ts` (183 LOC) — separate web-only method schemas | **Deleted** — merged into `client-requests.ts` |
| `DILIGENT_WEB_REQUEST_METHODS` (7 methods) — separate constant namespace | **Deleted** — unified into `DILIGENT_CLIENT_REQUEST_METHODS` |

Key architectural properties:
- **One protocol boundary**: All frontends (CLI, Web, Desktop) go through `DiligentAppServer.connect(connectionId, peer)`. The `RpcPeer` interface (`send` + `onMessage`) is the only transport abstraction.
- **Multi-connection built-in**: Thread subscriptions, notification routing, server request fan-out, turn initiator tracking — all in one place with consistent semantics.
- **Factory pattern**: `createAppServerConfig()` in `factory.ts` (97 LOC) eliminates web/CLI config duplication.
- **CLI parity**: CLI now uses `DiligentAppServer` via stdio framed transport (`app-server-stdio.ts`, `rpc-framed-client.ts`), same code path as web WebSocket.

This change resolves the two deepest findings from previous reviews simultaneously:
- **Shadow protocol elimination** (completed in prior reviews) is now architectural — there is no web-only request path.
- **rpc-bridge `as` casts** (3 reviews, growing from 7→9) — eliminated entirely. `DiligentAppServer.dispatchClientRequest()` uses protocol-typed `DiligentClientRequest` directly.

## Cross-Package Coherence

### 11. Protocol Method Unification — Complete
**Scope**: All packages
**Impact**: Eliminates an entire category of drift

`DILIGENT_WEB_REQUEST_METHODS` has been eliminated. All 22 client request methods are now under `DILIGENT_CLIENT_REQUEST_METHODS` in `packages/protocol/src/methods.ts`. The web-only methods (`config/set`, `auth/*`, `thread/subscribe`, `thread/unsubscribe`, `image/upload`) are first-class protocol citizens — CLI and Desktop can use them identically.

`packages/protocol/src/web-requests.ts` (183 LOC) has been deleted. Its Zod schemas (`ConfigSetParamsSchema`, `AuthSetParamsSchema`, `AuthRemoveParamsSchema`, `ModelInfoSchema`, etc.) were merged into `client-requests.ts`.

**Zero raw method strings remain in source code**. The only string literals are in `rpc-client.ts:26-34` where a `WebMethod` type union uses string literal types for TypeScript narrowing — these are compile-time only, not runtime values.

### 12. Type Flow — Clean, Single Source of Truth
**Scope**: All packages
**Impact**: No type duplication across packages

Type flow is now unidirectional: `protocol → core → cli/web/desktop`:

- **ModelInfo**: Defined once in `@diligent/protocol`, imported by all consumers. `ws-protocol.ts` is 4 lines (re-export only).
- **ProviderName**: `ProviderNameSchema` in protocol, `Record<ProviderName, ...>` patterns in core.
- **Method constants**: Single namespace per direction (client requests, server notifications, server requests).
- **Zod schemas**: All request params validated in `DiligentClientRequestSchema.safeParse()` inside `DiligentAppServer.handleRequest()` — one validation point.

### 13. Transport Parity — Achieved
**Scope**: packages/cli, packages/web, apps/desktop
**Impact**: All frontends use identical protocol path

| Frontend | Transport | Protocol Entry Point |
|----------|-----------|---------------------|
| Web | WebSocket (`Bun.serve` → `createWsPeer`) | `DiligentAppServer.connect()` |
| CLI | stdio framed transport (`rpc-framed-client.ts`) | `DiligentAppServer.connect()` |
| Desktop | Tauri sidecar → WebSocket | Same as Web |

The CLI now creates an in-process `DiligentAppServer` (via `app-server-stdio.ts`) or spawns a separate process with framed JSON-RPC over stdio. Both paths converge at `DiligentAppServer.connect()`. This means:
- Auth flows, tool settings, image upload — all available to CLI
- Multi-client subscriptions work across transports
- One set of tests covers all transports

## Forward Compatibility

### 15. server.ts as Universal Protocol Boundary — Sustainable Pattern
**Horizon**: D093/D094 will add ~6 notification types + capability negotiation

The `DiligentAppServer` pattern of `connect()` → `handleRequest()` → typed dispatch → `emit()` is extensible:
- Adding a new request method: add case to `dispatchClientRequest()`, add schema to `client-requests.ts`
- Adding a new notification: add to `DILIGENT_SERVER_NOTIFICATION_METHODS`, add to `event-mapper.ts`
- Adding a new transport: implement `RpcPeer` (2 methods)

D093 (fine-grained streaming deltas) will add ~6 new event types to `event-mapper.ts` and reducer cases to `thread-store.ts`. D094 (capability negotiation) will add a `streamingDeltaVersion` field to `INITIALIZE` params/response. Both fit cleanly into the existing patterns.

The main risk is server.ts LOC growth — see Finding #6.

## Summary

This review period represents the most significant architectural improvement in the project's history. The unified transport architecture (P036–P039) eliminated the web server's `rpc-bridge.ts` (864 LOC) and consolidated all protocol handling into `DiligentAppServer`, creating a single protocol boundary for all frontends. Simultaneously, 4 of the longest-standing persistent issues were resolved:

**Resolved persistent issues:**
- **Provider if/else chain** (7 reviews, since 2026-03-03): `PROVIDER_FACTORIES: Record<ProviderName, ...>` with explicit error (42d0642)

- **rpc-bridge `as` casts** (3 reviews, since 2026-03-05): file eliminated (e5a29b9)
- **Raw method strings** (4 reviews, since 2026-03-04): all source code uses protocol constants

**Key observations:**
- **Protocol unification**: `DILIGENT_WEB_REQUEST_METHODS` eliminated, all methods under one namespace
- **Transport parity**: CLI and Web both use `DiligentAppServer.connect()` — identical code path
- **ws-protocol.ts collapsed**: 90 LOC → 4 LOC (pure re-export)
- **Test growth**: 1,009 tests (+85), 2,493 assertions (+315), 105 files (+12)
- **E2E infrastructure**: `ProtocolTestClient`, `server-factory.ts`, `fake-stream.ts` — proper test framework

**Remaining concerns:**
- `server.ts` grew to 1,437 LOC — well-structured internally but will need decomposition when D093 arrives
- Deprecated backward-compat API still present (trivial cleanup)
- Version fallback `"0.0.1"` persists (8 reviews, cosmetic)

The architecture has reached a qualitative inflection point. Before this period, the protocol boundary was split across packages (core server + web rpc-bridge) with a separate "web-only" method namespace. Now there is exactly one protocol boundary, one method namespace, and thin transport adapters. This is the architecture the project was evolving toward — and it arrived through deliberate migration rather than accidental convergence.
