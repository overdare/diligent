## Structural Integrity

### 1. Core-Runtime Split — Major Architectural Milestone
**Impact**: Positive — resolves the most significant structural concern in the project's history

The core-runtime split (438aac2 + a029bd9) is the defining event of this period. Before the split, `packages/core` contained both reusable engine concerns (agent loop, providers, tool interfaces) and Diligent-specific runtime concerns (app-server, sessions, built-in tools, config, knowledge, skills, approval, collab). This meant any project wanting to reuse the agent engine would inherit the full Diligent runtime.

**After the split:**
- `packages/core` (45 source files): Pure agent engine — provider abstraction, agent loop, tool interfaces, auth primitives, EventStream
- `packages/runtime` (92 source files): Diligent runtime — app-server, sessions, tools, config, knowledge, skills, approval, collab, RPC

The package boundary is clean: core has **zero imports** from runtime or protocol. Runtime imports from core (117 locations across 67 files) — the expected one-way dependency.

Key file LOC changes during migration:

|------|-------------------|---------------------|--------|
| manager.ts | 909 | 777 | -15% |
| server.ts | 1,019 | 937 | -8% |
| thread-handlers.ts | 575 | 548 | -5% |
| provider-manager.ts | — | 237 | Simplified (was ~300+) |

The LOC decreases came from refactoring during migration — prompt options simplification (51081fc), provider auth injection (c5ae8ec), and ChatGPT provider split (44458ca) all took advantage of the migration to clean up interfaces.

### 2. Provider Architecture Cleanup — Three Coordinated Refactors
**Impact**: Positive — reduces provider addition surface area complexity

Three refactors improved the provider subsystem:

1. **ChatGPT split from OpenAI** (44458ca): ChatGPT OAuth (raw fetch to chatgpt.com) is now a separate provider from OpenAI (SDK-based Responses API). Previously they shared the "openai" provider name with branching logic. Now `ProviderNameSchema = z.enum(["anthropic", "openai", "chatgpt", "gemini"])` — each provider has a clean 1:1 mapping to its stream function.

2. **Provider auth injection** (c5ae8ec): `ProviderManager` no longer reaches into auth stores directly. Instead, auth bindings are injected via `ExternalProviderAuth` interface. `packages/runtime/src/auth/provider-auth.ts` (59 LOC) bridges the runtime's auth store to core's provider manager. This decouples core from runtime's auth persistence.

3. **Prompt options simplification** (51081fc): Streamlined the `StreamFunction` interface and session wiring. 37 files touched, net -14 LOC — a rare refactor that reduces total code.

Provider addition surface area remains bounded at 4 files, unchanged from previous review.

## Novel Perspective

### The Core-Runtime Split Created a Hidden Third Consumer of Provider Interfaces

- **Why no one has seen this**: All previous reviews tracked the provider interface as consumed by two actors: the agent loop (which calls `StreamFunction`) and the frontends (which display provider status). The core-runtime split introduced a third consumer that doesn't call the stream function but needs intimate knowledge of provider behavior: the **runtime auth binding layer**.

- **Evidence**:
  1. `packages/runtime/src/auth/provider-auth.ts` (59 LOC) implements `ExternalProviderAuth` for ChatGPT OAuth. It must know: when tokens expire, how to refresh them, that refresh tokens are single-use with rotation, and that concurrent refresh requests must be serialized with a lock. This is provider-specific behavioral knowledge living outside the provider file.

  2. `packages/runtime/src/app-server/config-handlers.ts:167-189` handles `auth/set` and `auth/remove` RPC calls. It must know which providers use API keys vs. OAuth tokens to route the auth operation correctly. The `ProviderName` type is insufficient — it doesn't encode the auth mechanism.

  3. `packages/runtime/src/auth/auth-store.ts` persists both API keys and OAuth tokens. Its storage schema (`AuthData.providers[name]`) must accommodate both auth mechanisms, but the type doesn't discriminate — it stores `{ apiKey?: string, accessToken?: string, refreshToken?: string }` for all providers regardless of which fields are meaningful.

  4. `packages/core/src/llm/provider-manager.ts:50-80` resolves auth at stream-call time through the injected binding, but the binding interface (`ExternalProviderAuth`) is ChatGPT-specific (token refresh, PKCE). If a 5th provider uses a different auth mechanism (e.g., service account JSON, API key rotation), the binding interface would need to change or a second binding type would be needed.

  Before the split, all of this knowledge was collocated in core's auth and provider directories. The split distributed it across core (interface), runtime/auth (implementation), and runtime/app-server (routing). This is correct architecturally — but it means adding a provider with a non-standard auth flow now requires changes in **3 packages** (protocol for the name, core for the stream function, runtime for the auth binding) instead of 2.

- **Implication**: The current `ExternalProviderAuth` interface is implicitly ChatGPT-shaped. Before adding a 5th provider, evaluate whether the auth binding interface should be generalized to a `ProviderAuthBinding` with a discriminated union (`{ type: "api-key" } | { type: "oauth", ... } | { type: "service-account", ... }`). This would make the provider addition surface area explicit about auth rather than leaving it as "implement ExternalProviderAuth and figure out which methods matter for your provider."

  This is not urgent — the current 4 providers all work. But it's the kind of assumption that becomes expensive to fix after 2-3 more providers are added with ad-hoc auth binding implementations.
