## Cross-Package Coherence

### Three Reducer Implementations Drift on Subtle Semantics
**Scope**: protocol, web, cli, vscode-extension

The three active reducers handle the same core events differently:

| Behavior | Web | CLI | VS Code |
|----------|-----|-----|---------|
| Thinking duration tracking | `activeReasoningStartedAt` + `activeReasoningDurationMs` | `thinkingStartTime` | Not tracked |
| Message end finalization | Clears in-flight items, finalizes reasoning | Converts to CLI item, finalizes thinking | Simple clear |
| Error recovery | Clears streaming items + live state | Delegated to CLI handlers | Sets `lastError` only |
| Collab events | Full handling (spawn, wait, close) | Not handled in reducer | Not handled |
| Side effects | None (pure state) | Returns `ThreadEventReducerEffect[]` | None (pure state) |

These differences are partly intentional (CLI needs effects for TUI rendering) but partly accidental (VS Code omitting thinking duration, error recovery not settling streaming items). Without the shared reducer as a baseline, there is no way to distinguish intended client-specific behavior from drift.

## Forward Compatibility

### Shared Reducer Adoption Is Now the Critical Path
**Horizon**: Breaks when the next event type is added (e.g., micro-compaction progress, thread fork notifications)

Adding a new `AgentEvent` type today requires editing:
1. `packages/protocol/src/conversation-reducer.ts` (shared, unused)
2. `packages/web/src/client/lib/thread-store.ts` (web reducer)
3. `packages/cli/src/tui/components/thread-event-reducer.ts` (CLI reducer)
4. `apps/vscode-extension/src/views/webview/state.ts` (VS Code reducer)

With the shared reducer adopted, this would be:
1. `packages/protocol/src/conversation-reducer.ts` (shared baseline)
2. Per-client: only if client-specific behavior is needed (effects, timing, etc.)

The cheaper fix is to adopt the shared reducer in VS Code first (lowest complexity, 147 LOC to replace), then web, then CLI. VS Code's `applyAgentEvent()` is almost a subset of the shared reducer already.

## Novel Perspective

### The Write-Only Module Anti-Pattern
- **Why no one has seen this**: The shared reducer was the recommended solution, so implementing it felt like progress. The commit message ("prototype shared AgentEvent state reducer") correctly signals it was a prototype. But no follow-up adoption work was scoped — the prototype shipped in isolation without a migration path for any client.
- **Evidence**: `packages/protocol/src/conversation-reducer.ts` is exported from `@diligent/protocol` index (line 3), has a lint fix commit (73cc5ac), but `grep` across all consumers returns zero imports. Meanwhile, the three client reducers continued receiving feature work (steering images in web, spacing fix in CLI) that was never reflected in the shared module.
- **Implication**: Shared infrastructure that ships without at least one real consumer is not shared infrastructure — it is a specification that may already be wrong. The project should adopt a "first consumer" rule: no shared module merges without at least one client importing it. For `conversation-reducer.ts` specifically, the VS Code extension is the ideal first adopter because its reducer is the smallest (147 LOC) and closest to the shared module's scope. Migrating VS Code would simultaneously validate the shared API and reduce VS Code's maintenance surface.
