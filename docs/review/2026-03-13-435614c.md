### 2. TUI Session Resume — Feature Parity Gap with Web
**Impact**: Architectural concern — TUI implements resume differently from web

The `--resume` feature (86ff75d) adds conversation history hydration to the TUI. The implementation in `hydrateThreadHistory()` (app.ts:717-748) manually parses `TranscriptEntry` objects and renders them to the chat view:

```
entry.type === "compaction" → addLines() with summary
entry.type === "message" && role === "user" → addUserMessage(text)
entry.type === "message" && role === "assistant" → addAssistantMessage(text)
```

The web client has equivalent functionality in `thread-hydration.ts` (439 LOC) — a dedicated module that handles the same `TranscriptEntry` types with richer rendering (tool results, thinking blocks, child sessions). The TUI's `hydrateThreadHistory()` is a simplified version that only renders text content from user and assistant messages, skipping tool calls, thinking blocks, and images.

This is functionally acceptable (TUI has limited rendering capability), but creates a **semantic divergence**: the web and TUI interpret the same protocol response (`thread/read` with transcript) differently, with no shared parsing logic. The transcript entry type union is consumed by two independent parsers with different coverage. If a new `TranscriptEntry` variant is added to the protocol, both parsers need updating independently.

## Novel Perspective

### The TUI Is Building a Shadow Client Architecture Without Shared Client Infrastructure

- **Why no one has seen this**: Previous reviews focused on `packages/protocol/` as the single source of truth and verified type imports flow correctly. Type-level compliance was confirmed — both clients import from `@diligent/protocol`. But type compliance is necessary, not sufficient. The deeper question is whether both clients share **behavioral interpretation** of protocol responses, not just their type definitions.

- **Evidence**:
  1. **Transcript parsing**: The web client has `thread-hydration.ts` (439 LOC) — a dedicated module that converts `ThreadReadResponse` into renderable state, handling all `TranscriptEntry` variants (message, compaction, tool_result, thinking, child_session). The TUI's `hydrateThreadHistory()` (app.ts:717-748, 31 LOC) handles only 2 of these variants (message, compaction). Both consume the same protocol response but with 14:1 LOC ratio and different coverage.

  2. **Notification adaptation**: The web client routes `DiligentServerNotification` events through `thread-store.ts` (854 LOC) — a zustand store that maintains a complete thread state model. The TUI routes the same notifications through `ProtocolNotificationAdapter.toAgentEvents()` (from `@diligent/core`) which converts to `AgentEvent` objects, then through `handleAgentEvent()` in app.ts. Different adaptation layers, different state models, for the same protocol events.

  3. **RPC method invocation**: The web client uses `use-thread-manager.ts` (204 LOC) for thread CRUD operations. The TUI uses `thread-manager.ts` (103 LOC) for the same operations. Both call the same RPC methods with the same parameters, but the wrappers are independently implemented with different error handling and state update patterns.

  There is no `packages/client-common/` or shared client-side interpretation layer. The protocol package defines the **wire format** (Zod schemas, method constants), but not the **client-side behavioral contract** (how to parse a transcript, what to do when a notification arrives, how to handle RPC errors).

- **Implication**: Today, with one developer maintaining both clients, the divergence is manageable. But the gap compounds:
  - Every new `TranscriptEntry` variant requires updating two parsers (439 LOC + 31 LOC) independently
  - Every new notification type requires updating two routing paths (thread-store.ts + app.ts handleServerNotification) independently
  - Every new RPC method requires updating two thread managers (204 LOC + 103 LOC) independently

  The fix is NOT to create a shared client library (the rendering targets are too different — React state vs terminal output). The fix is narrower: extract **protocol interpretation utilities** into `@diligent/protocol` or `@diligent/core/client`:
  - `parseTranscript(entries: TranscriptEntry[]): ParsedTranscript` — canonical parsing of all entry types, returns a renderer-agnostic structure
  - `categorizeNotification(notification: DiligentServerNotification): NotificationCategory` — classify notifications by domain (thread, turn, error, auth) so routing logic doesn't need to hardcode method names

  This would reduce the TUI's transcript parser to `const parsed = parseTranscript(entries); for (const item of parsed.items) { renderItem(item); }` — and the web's thread-hydration.ts would consume the same parsed structure. New entry types would be added in one place.
