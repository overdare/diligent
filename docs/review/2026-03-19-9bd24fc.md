## Novel Perspective

### The Debug-Viewer's Intentional Type Duplication Reveals an Unacknowledged Package Topology Decision

- **Why no one has seen this**: Previous reviews tracked type duplication as a defect — something to unify. The debug-viewer's types.ts (line 1: "Convention-based types — duplicated from @diligent/core by convention, NOT imported (DV-01)") asserts duplication as a feature, not a bug. This contradicts the project's otherwise strict type-flow principle (protocol → core → runtime → cli/web). No decision document records why this exception exists or when it should be applied to other packages.

- **Evidence**:
  1. **The debug-viewer has zero monorepo dependencies**. Its `package.json` imports only React, marked, and react-json-view-lite. It doesn't import `@diligent/protocol`, `@diligent/core`, or `@diligent/runtime`. This is unique — every other package in the monorepo participates in the type-flow chain.

  2. **The DV-01 convention is undocumented outside the file itself**. No entry in `decisions.md`, no mention in `ARCHITECTURE.md`, no plan document explaining the trade-off. A future contributor (or AI session) encountering `shared/types.ts` has no guidance on whether to follow this pattern for new standalone packages or to integrate with the monorepo's type chain.

  3. **The 8 `as unknown as` casts in parser.ts are a direct consequence** of this duplication. The parser reads JSONL files produced by `@diligent/runtime/session/persistence.ts`, which uses core's types. But the parser casts to debug-viewer's duplicated types. If core's `TranscriptEntry` gains a field, debug-viewer's copy won't have it — the cast silently drops the field. This is the same "silent fork" pattern that caused the notification adapter divergence, but in a different topology.

  4. **The real question is: should the debug-viewer be a monorepo package at all?** It sits in `packages/` alongside tightly-integrated packages but behaves as an external tool. This creates a category error: monorepo packages share types via imports; external tools duplicate types by convention. The debug-viewer is doing both — living in the monorepo but not participating in its type system.

  5. **The pattern has already spread**: 4 feature commits in this period added ~400+ LOC to the debug-viewer (Windows exe, dashboard, error parsing, thread preview). Each new feature that consumes session data adds more surface area to the duplicated type contract. The parser now handles 8+ entry types with cast-based bridging.

- **Implication**: The project needs to make an explicit decision (a `D095` candidate) about the debug-viewer's package topology:

  **Option A**: Keep it standalone, move it out of `packages/` to `tools/debug-viewer/`, and accept that its types are a snapshot that may drift. Add a version tag to the duplicated types and a CI check that compares the tag to core's type hash.

  **Option B**: Integrate it into the monorepo type chain — add `@diligent/protocol` as a dependency, replace the duplicated types, and remove the 8 casts in parser.ts. The debug-viewer gains type safety and loses deployment independence.

  **Option C** (the actual current state): Live in the monorepo, duplicate types, and hope the casts don't break. This is the default that accumulated without a decision. It works today because the session JSONL format is stable, but it will break silently when the format evolves.

  The cost of deciding now is ~30 minutes of writing a decision document. The cost of not deciding is proportional to how often the session entry format changes — and with the P056 items-only cutover just completed, the format is actively evolving.
