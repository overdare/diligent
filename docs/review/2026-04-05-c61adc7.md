## Forward Compatibility

### P061 will test the core/runtime boundary under real cross-package pressure
**Horizon**: When P061 implementation begins — affects protocol, core (4 provider files), runtime (3+ files), web, cli

P061 is the first plan since the core-runtime split (commit 438aac2) that requires coordinated changes across protocol → core → runtime → both frontends. The plan is well-structured with explicit file manifests and normalized contracts. Key architectural risk: the normalized Diligent web tool definitions in `core/src/llm/types.ts` must be rich enough to express 3 providers' native capabilities without leaking provider-specific semantics. The existing `NativeCompactionLookup` pattern provides a precedent, but web tools carry richer payloads (citations, sources, page content) than compaction results.

The plan's explicit non-goal — "No generalized provider-native tool framework" — is the right scope constraint for now but will need revisiting if future provider-native capabilities (code execution, file generation) follow similar patterns.

**Evidence**: `docs/plan/feature/P061-provider-native-websearch-webfetch.md`, scope table at lines 92-118

### Plugin hook discovery still runs per-turn — no caching
**Horizon**: Same — breaks when plugin count grows

`collectPluginHooks()` (plugin-loader.ts:57-92) still reimports and scans all plugins on every call. With 4 Overdare plugins, the overhead is tolerable but grows linearly. The function calls `importPluginModule()` which triggers dynamic `import()` on each invocation — Bun caches modules, so the actual I/O cost is minimal, but the hook collection logic itself (iterating configs, checking exports) runs fresh every time.

**Evidence**: `packages/runtime/src/tools/plugin-loader.ts:57-92`

### render-strategies RENDER_STRATEGIES map: static, not extensible by plugins
**Horizon**: Compounds when plugins need custom render strategies

The `RENDER_STRATEGIES` map (render-strategies.ts:32-206) contains 10 hardcoded entries (apply_patch, update_knowledge, search_knowledge, plan, skill, request_user_input, read, write, bash). Plugin-authored tools cannot register custom render strategies — they must build `ToolRenderPayload` directly. P061's web tools (`websearch`, `webfetch`) will need entries in this map, growing it to 12. This is fine for built-in tools, but the studiorpc plugin's own render.ts (359 LOC) duplicates summary/truncation patterns because it cannot participate in the strategy system.

**Evidence**: `packages/runtime/src/tools/render-strategies.ts:32-206`, `thirdparty/overdare/plugins/plugin-studiorpc/src/render.ts`

### E2E test foundation: 8 files, 1,278 LOC — solid but no web tool coverage path
**Horizon**: P061 implementation will need either new e2e tests or explicit gap acceptance

The e2e suite covers: conversation, custom agents, mode/config, multi-connection, protocol lifecycle, session resume, turn execution, WebSocket transport. Provider-native web tools would be the first feature that requires mocking provider-specific response shapes in e2e tests — current tests use mock providers but don't exercise provider-specific response block types. P061's test plan should address whether e2e coverage of web result blocks is in scope or explicitly deferred.

**Evidence**: `packages/e2e/` (8 test files, 1,278 LOC total)

## Novel Perspective

### The Remediation-as-Warm-Up Hypothesis Was Validated — Now Watch for Over-Correction

**Why no one has seen this**: The 2026-04-04 review identified a structural frequency problem where remediation only happened as a byproduct of feature delivery. This period's 100% remediation rate appears to validate the "batch Priority Actions into the start of each code session" recommendation. But this creates a new, subtler risk: the session produced 4 standalone remediation PRs and 0 feature PRs. The pendulum has swung from "remediation only with features" to "remediation without features."

**Evidence**:

1. **All 4 PRs are remediation-only**: PRs #79, #80, #81, #82 each address exactly one Priority Action with no feature content. This is the cleanest remediation session in review history — and also the first session with zero feature delivery since reviews began tracking.

2. **The combined LOC profile reveals the asymmetry**: The 4 remediation PRs changed ~90 net LOC of source code (render-strategies.ts is a 273 LOC extraction, not 273 LOC of new logic; runner.ts gained 29 LOC; plugin-sdk gained 15 LOC; package.json gained 1 LOC). P061 produced 645 LOC of planning markdown. The ratio is ~7:1 planning-to-implementation. This is not a problem in isolation, but it means the project's forward velocity is currently in documentation, not in shipped capability.

3. **P061 is the most cross-package plan since the core-runtime split**: It touches protocol, core (4 provider files + agent + types), runtime (tools + session + app-server), web, and cli. Implementation will be the first real stress test of whether the current package boundaries hold under coordinated multi-package change. The plan is well-structured, but its 645 LOC of specification will need to translate into a comparable volume of carefully sequenced implementation.

4. **The backlog is growing faster than it's shrinking**: P060 (backlog, 515 LOC, 2026-04-04) and P061 (backlog, 645 LOC, 2026-04-05) represent ~1,160 LOC of planned-but-unimplemented work queued in 2 days. Both plans are substantial cross-package changes. If the next session is also planning-focused, the ratio of planned-to-implemented work will tilt further toward planning debt.

5. **The remediation pipeline is now empty for the first time**: With all 4 Priority Actions resolved, there are no carried-forward remediation tasks. This is healthy — it means the next session can be entirely feature-focused without remediation drag. The risk is that without Priority Actions as "warm-up" tasks, the next session might lack the structured entry point that made this session's clean remediation possible.

**Implication**: The next 1-2 sessions should prioritize implementation over planning. P060 and P061 are both well-specified and ready for execution. Starting P061 implementation (specifically Task 1: normalized contracts in protocol + core types) would validate the plan's architectural assumptions early and produce the cross-package pressure that reveals whether the current boundaries are adequate. The empty remediation pipeline is an opportunity — use it for feature velocity, not more planning.
