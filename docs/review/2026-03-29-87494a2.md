### Desktop build-time plugin bundling — parallel loading path
**Horizon**: Breaks when plugin format or discovery changes

The desktop packaging pipeline (`apps/desktop/scripts/lib/defaults.ts`) bundles plugins at build time via `createPluginBundlePlan`. Runtime discovers and loads plugins via `packages/runtime/src/tools/plugin-loader.ts` (298 LOC). These are two independent paths with no shared contract validation:
- Build-time: copies source, runs Bun build, writes output package.json
- Runtime: discovers plugin directories, loads package.json, imports entry point

If the runtime plugin loader changes its discovery logic (directory structure, package.json shape, entry point convention), build-time bundling must be updated manually. Currently the only coupling is the `createPluginBundlePlan` function used by defaults.ts, which does reference plugin-bundle.ts. The risk is low today with a single plugin author, but would compound with third-party plugins.

**Evidence**: `apps/desktop/scripts/lib/defaults.ts:63-97`, `packages/runtime/src/tools/plugin-loader.ts`

## Novel Perspective

### thread/read Has Silently Become a Hybrid Persistence+Live-State Query

**Why no one has seen this**: Previous reviews tracked `thread/read` as a persistence operation — read JSONL, build transcript, return snapshot. The live collab overlay (commit 1763cf0) was reviewed as a "desktop notification fix" that happened to overlay agent statuses. No review has examined the architectural consequence: `thread/read` is no longer a pure persistence read.

**Evidence**:

1. **Before 1763cf0**: `thread/read` handler at `thread-handlers.ts:418-480` called `manager.getTranscript()`, built ThreadItem[] from persisted entries, and returned the snapshot. The data source was exclusively JSONL persistence. Any consumer calling `thread/read` twice with no intervening writes would get identical results.

2. **After 1763cf0**: `thread/read` calls `applyLiveCollabStatusesToSnapshot()` (line 434) which queries `runtime.agent?.registry?.getKnownAgents()` (line 46). This reads in-memory agent registry state — statuses that are never persisted to JSONL until an agent completes. Two identical `thread/read` calls can now return different results if an agent's status changed between calls.

3. **The idempotency contract changed without protocol documentation**: The protocol defines `thread/read` as a request that returns `ThreadReadResponse`. Nothing in the protocol schema or D096 (protocol version) indicates that thread/read results may include non-persisted live state. A client caching `thread/read` responses (e.g., the web client's `childThreadCacheRef` in App.tsx:74) may serve stale collab statuses.

4. **The live overlay is graceful but invisible**: The function uses `??` coalescing — it only fills fields that the persisted snapshot left empty. A spawn item with no live agent data passes through unchanged. This is correct behavior, but means the enrichment is invisible to anyone reading the protocol contract or the session JSONL. The "real" thread state is now the union of persistence + in-memory registry, with no single source of truth.

5. **The precedent matters more than the current instance**: The collab status overlay is 63 LOC and well-bounded. The risk is not this specific enrichment — it's the pattern. If future features add more live-state overlays (e.g., "currently streaming" indicators, pending approval status, child thread progress), `thread/read` would gradually become a real-time view query rather than a persistence snapshot, with no mechanism to distinguish cached from live data on the client.

**Implication**: Add a lightweight documentation marker — either a comment in the `thread/read` handler noting the hybrid nature, or a protocol-level field like `snapshot_at` timestamp that lets clients detect stale cached responses. The current behavior is correct, but the implicit contract change should be made explicit before additional live-state overlays are added.
