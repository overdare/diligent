## Cross-Package Coherence

### Plugin-SDK and Core Tool types: intentional divergence, undocumented contract boundary
**Scope**: packages/core, packages/plugin-sdk

Plugin-SDK and Core define independent `Tool`, `ToolContext`, and `ToolResult` interfaces with structural differences:

| Type | Core | Plugin-SDK | Divergence |
|------|------|-----------|------------|
| `ToolContext` | `toolCallId`, `signal`, `abort`, `onUpdate` | Same + `approve()`, `ask()` | Plugin gets more capabilities |
| `ToolResult` | `output`, `render`, `abortRequested`, `truncateDirection`, `metadata` | `output`, `render`, `metadata` | Core has internal control fields |
| `ToolRenderPayload` | `ToolRenderPayloadLike` (blocks: `unknown[]`) | `ToolRenderPayload` (blocks: `Array<Record<string, unknown>>`) | Different block typing |

The divergence is structurally intentional: plugins need `approve()`/`ask()` through ToolContext because they can't access runtime internals, while core tools use runtime bridges. Core's `abortRequested` and `truncateDirection` are internal control signals that shouldn't leak to plugins. However, no decision documents this boundary or the rules for when a new capability must appear in both interfaces.

**Evidence**: `packages/core/src/tool/types.ts:3-30`, `packages/plugin-sdk/src/index.ts:14-54`

## Novel Perspective

### The Plugin-SDK Is an Inverted Capability Contract — Plugins Get More ToolContext Than Core

**Why no one has seen this**: Previous reviews tracked type duplication as a coherence concern (e.g., ProviderName, Mode/ModeKind). The plugin-sdk was reviewed for stability and sufficiency for external authors. But no review examined the *direction* of the capability gap between plugin-sdk and core Tool interfaces.

**Evidence**:

1. **Plugin-SDK ToolContext** (`packages/plugin-sdk/src/index.ts:30-37`) exposes `approve()` and `ask()` methods. These let plugins request user approval and collect structured input during tool execution.

2. **Core ToolContext** (`packages/core/src/tool/types.ts:12-17`) has only `toolCallId`, `signal`, `abort()`, and `onUpdate()`. No `approve()` or `ask()`.

3. **This means plugins have capabilities that built-in tools lack at the type level.** Built-in tools (bash, read, glob, etc.) must access approval through runtime bridges wired outside the ToolContext interface. The approval engine in runtime intercepts tool execution and injects approval decisions — but the tool's own `ToolContext` doesn't expose this.

4. **The inversion matters for contributor mental models.** A contributor reading `packages/core/src/tool/types.ts` would conclude that tools cannot interact with users during execution. A contributor reading `packages/plugin-sdk/src/index.ts` would conclude they can. Both are correct — but for different tool populations with no documented boundary explaining why.

5. **Commit 008208d amplifies this.** When the overdare plugin needed new capabilities, changes landed in both `core/tool/types.ts` (+2 lines) and `plugin-sdk/src/index.ts` (+2 lines) simultaneously. Without a documented contract boundary, each such change is a judgment call about which interface to extend — and the judgment is currently made ad-hoc per commit.

**Implication**: Add a decision entry (D098) documenting the plugin-SDK / core ToolContext boundary: core ToolContext is the minimal execution contract (signal, abort, streaming updates); plugin-sdk ToolContext extends it with runtime-bridged capabilities (approve, ask) that built-in tools access through a different mechanism. This prevents future contributors from either (a) adding `approve()` to core ToolContext (breaking the abstraction) or (b) forgetting to add new capabilities to plugin-sdk (breaking plugin parity).
