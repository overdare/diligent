### Plugin hooks cross 5 packages — clean contract flow
**Scope**: packages/plugin-sdk, packages/runtime, packages/core (types only)

The hooks system touches 5 packages but maintains unidirectional contract flow:
- `plugin-sdk` exports `PluginHookInput` and `PluginHookResult` for external plugin authors
- `runtime/hooks/runner.ts` defines internal `HookInput`, `HookResult`, `PluginHookFn` types
- `runtime/config/schema.ts` defines hook configuration via Zod
- `runtime/tools/plugin-loader.ts` discovers plugin hooks at load time
- `core/types` provides `Message` types consumed by hook utilities

The SDK types (`PluginHookInput`/`PluginHookResult`) are intentionally not identical to the internal types (`HookInput`/`HookResult`). The SDK uses optional fields and an index signature; the internal types use strict required fields. This is a correct boundary decision — plugin authors get a lenient contract, while runtime operates on validated types.

**Risk**: `PluginHookFn` in `runner.ts:106` accepts `HookInput` (internal type), but plugins export functions typed against `PluginHookInput` (SDK type). Since `HookInput` has `[key: string]: unknown` and `PluginHookInput` also has `[key: string]: unknown`, these are structurally compatible. But if runtime adds a required field to `HookInput` that `PluginHookInput` doesn't have, the types will silently diverge. This is the same pattern D098 addressed for `ToolContext` — it may warrant a similar decision entry.

**Evidence**: `packages/plugin-sdk/src/index.ts:32-53`, `packages/runtime/src/hooks/runner.ts:15-29,106`

## Forward Compatibility

### server.ts crossed 1,000 LOC — hook orchestration as the next growth vector
**Horizon**: Compounds when new hook events are added (e.g., ToolExecution, SessionStart)

server.ts now holds `runStopHooksFor()` (lines 913-949), which orchestrates shell + plugin hook execution for Stop events. This method duplicates the hook dispatch pattern already present in thread-handlers.ts (lines 368-395) for UserPromptSubmit hooks. The pattern is:

```
1. Collect shell handlers from config
2. Collect plugin handlers via collectPluginHooks()
3. Run shell handlers, check for block
4. Run plugin handlers, check for block
5. Handle blocked/allowed result
```

Today this duplication exists in exactly 2 places. When new hook events are added (plausible: ToolExecution, SessionStart, Error), each will need the same pattern. Extracting a `runCombinedHooks(shellHandlers, pluginHandlers, input, cwd)` function into `hooks/runner.ts` would consolidate the pattern to one place and reduce server.ts by ~20 LOC.

**Evidence**: `packages/runtime/src/app-server/server.ts:913-949`, `packages/runtime/src/app-server/thread-handlers.ts:368-395`

### Plugin hook discovery runs per-turn — no caching
**Horizon**: Breaks when plugin count or discovery latency grows

`collectPluginHooks()` in `plugin-loader.ts` is called on every turn start (thread-handlers.ts:370) and every Stop event (server.ts:917). Each call scans enabled plugins, imports their modules, and extracts hook functions. With the current 2 plugins, this is fast. But plugin discovery involves file system operations and dynamic imports — if the plugin count grows to 10+, the per-turn overhead could become noticeable.

The existing `resolveTools()` path already has caching via tool catalog. Plugin hooks could benefit from similar caching — discover once per config change, reuse across turns.

**Evidence**: `packages/runtime/src/tools/plugin-loader.ts:57-92`, `packages/runtime/src/app-server/thread-handlers.ts:370`, `packages/runtime/src/app-server/server.ts:917`

### Plugin-SDK hook types lack versioning or capability negotiation
**Horizon**: Breaks when hook contract evolves

The SDK exports `PluginHookInput` with `[key: string]: unknown` — an open contract that allows new fields. But there's no mechanism for a plugin to declare which hook events it supports, what SDK version it targets, or what input fields it expects. If the `Stop` hook input shape changes (e.g., adding a `toolResults` field), old plugins silently ignore the new field, which is fine. But if the runtime starts expecting a new return field from plugins (e.g., `modifiedPrompt`), there's no way to distinguish a plugin that intentionally returned `undefined` from one that doesn't know about the field.

This is acceptable at the current stage (internal plugins only), but should be addressed before third-party plugin adoption. D098 could be extended to cover hook contract stability.

**Evidence**: `packages/plugin-sdk/src/index.ts:32-53`

## Novel Perspective

### The Hook Dispatch Pattern Creates an Implicit Execution Order Contract Between Server and Thread-Handlers

**Why no one has seen this**: Reviews have tracked the hooks system as a single cohesive feature. The implementation distributes hook dispatch across two files — but the distribution isn't arbitrary. It creates an implicit temporal contract: UserPromptSubmit hooks run synchronously in the thread-handler turn flow (blocking the turn before agent execution), while Stop hooks run in the server's turn-consumption callback (after agent completion, before the next turn can start). These two execution contexts have different error handling, different state visibility, and different reentrancy characteristics, but share the same `HookResult` contract.

**Evidence**:

1. **UserPromptSubmit in thread-handlers.ts:368-431**: Runs inside `handleTurnStart()`, after the user message is constructed but before the agent loop begins. If blocked, it manually resets runtime state (`abortController = null`, `isRunning = false`, `currentTurnId = null`) and emits ERROR + TURN_COMPLETED + THREAD_STATUS_CHANGED. This is a clean abort, but the state reset (lines 399-401) directly manipulates ThreadRuntime fields — it is the only place outside SessionManager that touches these fields during a turn lifecycle.

2. **Stop in server.ts:913-949**: Runs inside `runStopHooksFor()`, called from the `onChildStop` callback in collab registry and from the `onStop` callback in SessionManager config. When blocked, it returns `{ continueWith: userMessage }` which causes SessionManager to re-run the agent with the hook's reason as a user message. This is reentrant — the re-run triggers another full turn, which will trigger another Stop hook check with `isRerun = true`.

3. **The asymmetry matters**: UserPromptSubmit blocking is a terminal state (turn ends), but Stop blocking is a reentrant state (turn restarts). A plugin author implementing both hooks must understand that `onStop` with `blocked: true` causes their `onUserPromptSubmit` to NOT fire for the re-run turn — the re-run bypasses the normal `handleTurnStart` flow and goes directly through SessionManager's `runAgentLoop`. This is correct behavior (the re-run is system-initiated, not user-initiated), but it's invisible in the SDK documentation.

4. **The `stop_hook_active` field is the only guard against infinite loops**: When a Stop hook blocks, the re-run sets `stop_hook_active: true` in the hook input (server.ts:927). Plugins are expected to check this field and avoid re-blocking. But `stop_hook_active` is not a field in `PluginHookInput` — it arrives through the `[key: string]: unknown` index signature. A plugin author reading only the SDK types would not discover this field exists.

5. **Thread-handlers directly manipulates ThreadRuntime state on hook block**: Lines 399-401 set `runtime.abortController = null`, `runtime.isRunning = false`, `runtime.currentTurnId = null`. This is the same state that SessionManager normally manages through `runAgentLoop()` → `completeAgentLoop()`. If a future change adds cleanup logic to SessionManager's completion path, the hook-blocked code path will miss it.

**Implication**: (a) Add `stop_hook_active` as an explicit optional field in `PluginHookInput` rather than relying on the index signature — this makes the reentrancy guard discoverable. (b) Consider routing the hook-blocked abort through SessionManager's existing state management rather than directly resetting ThreadRuntime fields, to prevent the two abort paths from diverging.
