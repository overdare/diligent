### 2. tool-info.ts — Shadow Tool Registry in Web Client (509 LOC)
**Impact**: Compounds with every tool addition or parameter change

`packages/web/src/client/lib/tool-info.ts` has grown from a display-name mapping into a **parallel tool definition layer**. It now contains:

| Section | Lines | What it hardcodes |
|---------|-------|-------------------|
| `TOOL_MAP` | 11–39 | 28 tool name → display info mappings |
| `summarizeInput()` | 198–266 | 7 tool-specific parameter extraction branches |
| `summarizeOutput()` | 179–195 | 2 tool-specific output parsing branches |
| `deriveRenderPayload()` | 272–445 | 11 tool-specific reconstruction branches |
| `parsePatchForRender()` | 447–509 | Regex-based patch format parser |

`deriveRenderPayload()` reverse-engineers tool semantics from raw text — it parses JSON input to extract `file_path`, `old_string`, `new_string`, `command`, etc. by tool name. This is a **reconstruction engine** that duplicates knowledge from core tool definitions:
- `bash` → extracts `command` parameter (must match bash.ts schema)
- `edit` → extracts `file_path`, `old_string`, `new_string` (must match edit.ts schema)
- `apply_patch` → parses patch envelope format (must match apply-patch.ts format)
- `read` → extracts `file_path`, strips line numbers (must match read.ts output format)

Unlike type flow (TypeScript catches) or protocol compliance (Zod validates), this coupling is **invisible at compile time**. If a tool renames `file_path` to `path`, the web client silently degrades to generic text rendering with no error.

## Forward Compatibility

### 14. ToolRenderPayload — Good Architecture, Fragile Fallback
**Horizon**: Every tool addition or parameter change

The ToolRenderPayload system (615b487) has the right architecture: tools produce structured render blocks at execution time, and clients render block types generically. The `ToolRenderBlocks.tsx` (459 LOC) component dispatches purely on block type (`file`, `command`, `diff`, `list`, etc.) — zero tool-specific logic.

But `deriveRenderPayload()` in `tool-info.ts` exists as a **session hydration fallback** for historical sessions that lack render data. This function must know every tool's parameter schema to reconstruct render blocks from raw text. As the tool set grows (plugins, new builtins), this function becomes an unbounded maintenance liability. Currently it handles 11 tool names; each new tool that wants structured rendering in historical sessions needs a new branch.

**Mitigation path**: When render payloads are present (all new sessions), `deriveRenderPayload()` is never called. The function's scope is limited to pre-ToolRenderPayload sessions. Once those sessions age out, the function becomes dead code. No action needed now — but do not add new branches for new tools. Let unknown tools fall through to generic text rendering.

## Novel Perspective

### The Web Client Contains Two Tool Systems That Cannot See Each Other

- **Why no one has seen this**: Previous reviews tracked tool rendering as a frontend concern and tool definitions as a core concern. These were assessed independently — "are tools well-defined?" (yes) and "does the web render them?" (yes). No review has examined the **interface between these systems**, because there isn't one. That's the problem.

- **Evidence**: The project has two independent representations of what a tool is:

  1. **Core tool definitions** (`packages/core/src/tools/*.ts`): Each tool declares its name, Zod parameter schema, and `execute()` function. These are the canonical tool definitions. They produce `ToolRenderPayload` blocks at runtime.

  2. **Web tool knowledge** (`packages/web/src/client/lib/tool-info.ts`, 509 LOC): `TOOL_MAP` (28 entries), `summarizeInput()` (7 tool-specific branches), `deriveRenderPayload()` (11 tool-specific branches parsing JSON parameter names like `file_path`, `old_string`, `command`). This is a **second tool definition** embedded in the web client that knows tool parameter schemas by convention, not by import.

  These two systems have **zero shared interface**. The core tool registry (`packages/core/src/tool/registry.ts`) builds a `Map<string, Tool>` but exports no metadata that the web client consumes for rendering decisions. The web client's `TOOL_MAP` is maintained manually. When model-dependent tool loading (482f964) was added, `defaults.ts` gained `isOpenAIProvider()` branching — but `tool-info.ts` already had entries for both `edit` and `apply_patch` independently, with no awareness that they're alternatives.

  The protocol layer (`ToolDescriptorSchema` in `client-requests.ts`) provides tool state for the settings UI but not rendering metadata. The `ToolRenderPayload` in protocol provides block-type schemas but not tool-to-block-type mapping. The gap is: **no protocol-level "tool display contract"** — a tool's icon, category, summary format, and fallback rendering strategy are all encoded in the web client's local knowledge.

- **Implication**: Every new tool requires a manual `TOOL_MAP` entry in tool-info.ts, and if it wants structured historical rendering, a `deriveRenderPayload()` branch. Plugin tools get generic rendering (icon "⚙", no structured blocks) because the web client has no knowledge of them. This is tolerable for builtins (finite, stable set) but becomes a friction point as the plugin ecosystem grows — plugin authors have no way to declare display metadata. The fix is not to move tool-info.ts into protocol (over-engineering for current needs), but to **freeze `deriveRenderPayload()` at its current scope** — no new branches — and rely on the ToolRenderPayload system for all future tools. The TOOL_MAP display metadata is a separate, smaller concern that could eventually move to a tool manifest field.
