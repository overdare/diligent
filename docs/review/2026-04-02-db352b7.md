## Cross-Package Coherence

### Analytics plugin uses `Record<string, unknown>` instead of SDK types
**Scope**: packages/plugin-sdk, thirdparty/overdare/plugins/plugin-analytics

The analytics plugin's `onStop` function (plugin-analytics/src/index.ts:75) is typed as:
```typescript
async function onStop(input: Record<string, unknown>): Promise<Record<string, unknown>>
```

It does not import or reference `PluginHookInput` or `PluginHookResult` from `@diligent/plugin-sdk`. Instead, it uses `input.usage as UsageData`, `input.user_id as string`, `input.model as string` — all untyped field access with `as` casts. This is the first plugin hook implementation, and it bypasses the SDK types entirely.

The contract mismatch is: runtime calls plugin hooks with `HookInput` (runner.ts:106), which is structurally compatible with `PluginHookInput` via index signatures. But the analytics plugin doesn't even attempt to use `PluginHookInput`, suggesting the SDK types are insufficient for real plugin development. The SDK doesn't export `usage`, `model`, `provider`, or `user_id` as known fields — only `hook_event_name`, `session_id`, `transcript_path`, `cwd`, and `permission_mode`.

**Evidence**: `thirdparty/overdare/plugins/plugin-analytics/src/index.ts:75,83,88-96`, `packages/plugin-sdk/src/index.ts:32-39`

### Third JSONC parser implementation — analytics plugin adds a regex-based variant
**Scope**: thirdparty/overdare/plugins/plugin-analytics, apps/desktop/src-tauri

The analytics plugin (plugin-analytics/src/index.ts:37-39) implements JSONC comment stripping as:
```typescript
function stripJsonComments(text: string): string {
  return text.replace(/\/\/.*$/gm, "").replace(/\/\*[\s\S]*?\*\//g, "");
}
```

This is the third distinct JSONC parsing implementation in the repo:
1. **Rust** (update.rs): Character-level state machine respecting quoted strings (fixed in PR #67)
2. **Runtime**: Uses Bun's native JSONC support via `Bun.file().json()` or the config loader
3. **Analytics plugin**: Regex replacement that strips `//` comments globally, including inside quoted strings

The analytics plugin's implementation has the same bug that PR #67 fixed in the Rust parser — it will corrupt URLs or any quoted value containing `//`. This bug is currently latent because `overdare.jsonc` only contains simple key/value pairs, but it will trigger if the config ever contains an endpoint URL.

**Evidence**: `thirdparty/overdare/plugins/plugin-analytics/src/index.ts:37-39`, `apps/desktop/src-tauri/src/update.rs:84-116`

## Novel Perspective

### The Analytics Plugin Creates the First Transparency-Opaque Data Flow

**Why no one has seen this**: Previous reviews evaluated the plugin system through the lens of contract stability and type safety. The analytics plugin is the first plugin that initiates outbound network communication, and its fire-and-forget pattern (void fetch().catch()) means the data flow is invisible to every transparency mechanism in Diligent.

**Evidence**:

1. **The core principle is "Transparency over complexity"** — README.md states: "Without transparency, development becomes increasingly difficult. Keep things simple so every layer can be debugged, experimented with, measured, verified, and improved." Every other data flow in Diligent is observable: agent events flow through the protocol, tool executions are logged in session transcripts, RPC traffic is inspectable via debug-viewer.

2. **The analytics fetch is fire-and-forget with swallowed errors**: `plugin-analytics/src/index.ts:100-107` — `fetch(endpoint, {...}).catch(() => {})`. If the analytics endpoint is down, misconfigured, or sending malformed data, there is zero feedback anywhere in the system. No session transcript entry, no protocol notification, no debug-viewer event.

3. **The data includes user-identifying information**: The analytics record includes `userId`, `cwd` (basename), `sessionId`, `model`, and token counts (`plugin-analytics/src/index.ts:86-97`). This data flows to `https://mrgfwpdrtzvlnyogudnc.supabase.co` — a hardcoded external endpoint. Users have no in-product visibility that this data is being sent, no protocol notification when it fires, and no way to discover it happened after the fact (since errors are swallowed and successes produce no output).

4. **The test/CI guard is an implicit acknowledgment of the opacity**: Commit db352b7 added `shouldSkipAnalyticsSend()` which checks `NODE_ENV`, `CI`, and `DILIGENT_ANALYTICS_ALLOW_IN_TEST`. This guard exists because the analytics call has no mock/intercept point — it must be suppressed at the plugin level rather than at a transport or protocol layer. If the hook system had an observable side-effect channel, the test could assert on the side effect instead of suppressing it.

5. **The hook return value carries no signal that a side effect occurred**: `onStop` returns `{}` regardless of whether the fetch was attempted, succeeded, or failed. The runtime's `runPluginHooks()` receives an empty `Partial<HookResult>` and has no indication that the plugin made an external network call. Compare this to tools, which return `ToolResult` with output and metadata that become part of the session transcript.

**Implication**: Consider adding an optional `sideEffects` field to `PluginHookResult` (e.g., `sideEffects?: Array<{ type: string; target: string; status: "fired" | "skipped" }>`) that plugins can use to declare what external actions they took. This doesn't need to be enforced — plugins can ignore it — but it provides a transparency affordance for plugins that want to be good citizens. The runtime could log these to the session or surface them in debug-viewer. Without this, each new plugin that makes external calls further erodes the transparency guarantee that defines the product.
