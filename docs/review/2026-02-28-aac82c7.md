**YELLOW — Solid foundation, but two axes show yellow signals that need resolution before Phase 5.**

The Gemini provider itself is structurally clean and plugs into the D003 abstraction precisely as designed. The yellow signals come from two sources: (1) the commit period ends with 10 failing tests — a direct violation of the project's "run tests after code changes" rule; and (2) the Gemini addition reveals that the provider abstraction layer has 4+ hardcoded-provider locations that must all be updated in lockstep for each new provider, with the setup wizard already missing the Gemini update. ### Gemini Capability Flags Pattern: Yellow Trajectory

`supportsThinking?: boolean` and `defaultBudgetTokens?: number` on `Model` (`provider/types.ts:11-12`) are the start of per-provider optional flag accumulation. The semantic mismatch is already present: Gemini/Anthropic use a token count, OpenAI uses an enum effort level — and the implementation at `openai.ts:41` hardcodes `"high"` and discards the numeric budget value entirely. As more providers with distinct capability shapes are added, this flat optional field pattern will become increasingly hard to type-check accurately.

**Impact**: Low now. Risk is trajectory — Phase 5 MCP/multi-agent additions should prompt extracting a `capabilities` object before the flag count grows further.

### CLI Provider Hardcoding: Active Bug Found

The Gemini addition requires updating **4 locations** in the CLI:

| Location | Status |
|----------|--------|
| `config-writer.ts:13` — `saveApiKey` union type | Updated ✅ |
| `provider.ts:149-154` — `promptApiKey` hint chain | Updated ✅ |
| `provider-manager.ts:91-96` — `getOrCreateStream` if-else | Updated ✅ |
| `app.ts:218-221` — `wizardEnterApiKey` hint chain | **NOT updated ❌** |

**Active bug**: Users who select Gemini during first-run setup see the OpenAI URL (`https://platform.openai.com/api-keys`) and `sk-...` placeholder — directly wrong for Gemini. This is caused by `wizardEnterApiKey` in `app.ts` having its own copy of the hint logic that was not updated. The fix is 3 lines, but the root cause is that hint logic is duplicated in 2 places rather than driven by a provider registry.

**Impact**: Blocks first-run experience for Gemini users.

In-progress changes (model persistence, cancel UX, spinner fixes) are coherent and low-risk. One note: removing the Ctrl+C confirmation dialog at the session level may create UX tension when L4 approval dialogs arrive at the tool level in Phase 5. Not blocking.

### 2. Fix 10 failing tests (Test Suite Health)
**What**: Update `packages/core/test/config-loader.test.ts` expectations to match the new default model and `AGENTS.md` filename. Likely 3-5 line change.
**Why**: Broken test suite violates the "run tests after code changes" rule and creates noise that masks real regressions.

### 3. Add HTTP 529 handling to `classifyGeminiError` (Behavioral Defect)
**What**: Add a `529` case to `classifyGeminiError` in `packages/core/src/provider/gemini.ts:207-228` returning `{ errorType: "overloaded", isRetryable: true }`, matching `anthropic.ts:251-252` and `openai.ts:253-254`.
**Why**: Without this, Gemini service overload silently disables retry, causing spurious failures for users.
**Decision ref**: D010 (error classification must be provider-consistent)

### 4. Extract `isNetworkError` to shared provider util (DRY — Now 3 Copies)
**What**: Create `packages/core/src/provider/errors.ts` with shared `isNetworkError()`. Remove the 3 identical copies from `anthropic.ts:284`, `openai.ts:291`, `gemini.ts:235`.
**Why**: Three identical copies. A fourth provider would add a fourth. This is the forcing function — consolidate before Phase 5 adds dynamic provider registration.
**Decision ref**: Extends D010 (error classification infrastructure)

### 5. Drive provider hints from a registry (Forward Compat)
**What**: Create a `PROVIDER_METADATA` object (or extend `KNOWN_PROVIDERS` in `provider/models.ts`) with `apiKeyUrl` and `apiKeyPlaceholder` per provider. Replace the if-else chains in `provider.ts:149-154` and `app.ts:218-221` with a single lookup.
**Why**: The active wizard bug (Action 1) is a symptom of duplicated hint logic. The fix prevents the same class of bug when a 4th provider is added.

### 6. Remove speculative model IDs from `models.ts` (Accuracy)
**What**: Remove `gemini-3.1-pro-preview` and `gemini-3-flash-preview` from `provider/models.ts:63-76` until they exist in the Gemini API.
**Why**: Selecting these models produces a silent 404 error classified as `"unknown"` (non-retryable). The registry should only contain verified models.
