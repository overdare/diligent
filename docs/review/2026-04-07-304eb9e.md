## Cross-Package Coherence

### Thread-store state management asymmetry between web and CLI
**Scope**: packages/web, packages/cli

The thread-store — the central state container for conversation rendering — has diverged not just in rendering (the known persistent issue) but in state architecture:

| Concern | Web | CLI |
|---------|-----|-----|
| Thread state | `thread-store.ts` (641 LOC) | `thread-store.ts` (576 LOC) |
| Event reduction | Inline in thread-store | `thread-event-reducer.ts` (398 LOC) |
| Utility helpers | `thread-utils.ts` (embedded) | `thread-store-utils.ts` (568 LOC) |
| Collab handling | `collab-reducer.ts` (283 LOC) | Inline in thread-store-utils |
| Hydration | `thread-hydration.ts` (308 LOC) | `app-session-lifecycle.ts` (232 LOC) |
| **Total** | **~1,232 LOC** | **~1,774 LOC** |

The 542-LOC gap is partly because CLI must build its own render items (ANSI-formatted components) while web receives structured render payloads from the server. But the architectural split is different: web uses React reducers with a single thread-store file, while CLI uses a separate event-reducer pattern with extracted utility modules. New protocol events require understanding both patterns.

## Novel Perspective

### Tool Classification Metadata Lives in the Wrong Package

- **Why no one has seen this**: The `COLLAB_TOOL_NAMES` duplication in CLI has been treated as a one-off sync issue (documented at `thread-store-utils.ts:15`). Previous reviews noted it as a known design choice per D097/D098. The structural issue — that tool classification metadata is owned by `runtime` but needed by clients that deliberately don't import `runtime` — has not been identified as a package-boundary misplacement.

- **Evidence**:
  - `packages/runtime/src/tools/tool-metadata.ts` defines the `TOOL_CAPABILITIES` registry with 5 classification dimensions: `immutable`, `planModeAllowed`, `collabExcluded`, `hasCustomRender`, `executionMode`
  - This registry derives 4 exported sets: `IMMUTABLE_TOOLS`, `PLAN_MODE_ALLOWED_TOOLS`, `COLLAB_TOOL_NAMES`, `CUSTOM_RENDER_TOOLS`
  - CLI currently duplicates only `COLLAB_TOOL_NAMES` — but it already needs to distinguish tool types for rendering (collab events get special timeline rendering in `thread-store-utils.ts:364,417,478`)
  - Web does not need this duplication because it receives structured render payloads from the server (which has access to runtime)
  - The protocol package (`packages/protocol/`) already owns the tool name type (`tool-name.ts`, 19 LOC) and render payload schemas — it is the natural home for tool classification metadata
  - `packages/protocol/src/data-model.ts` (682 LOC) already defines `ToolRenderPayloadSchema` and tool-related data shapes — tool classification fits this scope

- **Implication**: If tool classification metadata (specifically the subset needed by clients: collab classification, custom render flag) were exported from `@diligent/protocol` instead of `@diligent/runtime`, the CLI could import it directly without either duplicating constants or taking a dependency on runtime. This would:
  1. Eliminate the manual sync fragility for `COLLAB_TOOL_NAMES`
  2. Prevent future duplications as CLI grows its event handling (e.g., if it needs `CUSTOM_RENDER_TOOLS` to decide rendering strategy)
  3. Reinforce protocol's role as the single source of truth for cross-package contracts
  4. Keep runtime as the owner of operational tool behavior (execution, approval, etc.) while sharing classification metadata through the contract layer

  The migration is small: move the `TOOL_CAPABILITIES` registry and its derived sets to protocol, have runtime re-export for backward compatibility. No behavioral changes required.

## Persistent Issues

### TUI-Web transcript parsing divergence
- **First raised**: 2026-03-13 (435614c)
- **Appearances**: 20+ reviews (435614c through 2026-04-07)
- **Status trajectory**: Same — stable at approximately 1,232 LOC (web) vs 1,774 LOC (CLI) for thread-store + rendering combined
- **Current evidence**: `packages/web/src/client/lib/thread-store.ts` (641 LOC), `packages/web/src/client/lib/thread-hydration.ts` (308 LOC), `packages/web/src/client/lib/collab-reducer.ts` (283 LOC) vs `packages/cli/src/tui/components/thread-store.ts` (576 LOC), `packages/cli/src/tui/components/thread-store-utils.ts` (568 LOC), `packages/cli/src/tui/components/thread-event-reducer.ts` (398 LOC), `packages/cli/src/tui/app-session-lifecycle.ts` (232 LOC)
- **Why it persists**: The divergence is architecturally stable. Web receives structured render payloads from the server and uses React state patterns; CLI builds its own render items from raw events using a custom reducer pipeline. New features (provider-native blocks, collab events) are added to both clients through their respective patterns without increasing the divergence ratio. The cost is tolerable: each new protocol event type requires implementation in both patterns, but the patterns are well-established and predictable.
