## Novel Perspective

### The Web Server Package Has Silently Acquired a Second Runtime Persona

**Why no one has seen this**: Previous reviews tracked the desktop sidecar as a thin wrapper over shared web/runtime behavior — which it is. The sidecar hardening commit (dfb0a50) was reviewed as a "desktop fix" because its commit message says "fix(desktop)". But the TypeScript-side changes landed in `packages/web/`, not `apps/desktop/`. No review has examined the cumulative effect: the web server now has two distinct runtime personas that are controlled by implicit launch-time arguments, not by explicit package boundaries.

**Evidence**:

1. **Persona 1: Standalone web server.** When launched without `--parent-pid`, `startParentWatchdog()` at `packages/web/src/server/index.ts:51-57` returns `null` and the server runs independently. No exit-on-parent-death behavior. This is the original design.

2. **Persona 2: Desktop sidecar child process.** When launched with `--parent-pid <pid>` (passed by Rust `sidecar.rs:spawn_updated_sidecar`), the web server starts a 2-second polling timer that terminates the process if the parent dies. The cleanup chain at lines 366-379 also handles SIGTERM/SIGINT for graceful shutdown. This persona is desktop-specific.

3. **The persona switch is invisible at the package boundary.** `packages/web/package.json` does not mention "sidecar" or "desktop". The `CreateServerOptions` interface (line 22-27) contains no sidecar-related fields. The sidecar-specific logic lives in the process bootstrap code (line 351-392), not in the `createWebServer` function itself. A contributor reading the `createWebServer` API would not discover the parent-watchdog behavior.

4. **The log file redirect is also a sidecar concern.** `args.logFile` (line 355-356) redirects stdout/stderr to a file when running as a sidecar. This is necessary because the sidecar's stdout is parsed by Rust for the `DILIGENT_PORT=` line, so log output must go elsewhere. Again, this is desktop-sidecar-specific behavior in the shared web package.

5. **The growth trajectory matters.** Today this is ~30 LOC of well-isolated bootstrap code. But the sidecar persona is likely to acquire more responsibilities: update notifications to the Rust shell, tray icon state reporting, window focus events, or sidecar health reporting. Each addition would further embed desktop-specific behavior in `packages/web/`.

**Implication**: Consider separating the sidecar bootstrap into a dedicated entry point — either `packages/web/src/server/sidecar-main.ts` that wraps `createWebServer` with sidecar-specific lifecycle, or a thin launcher in `apps/desktop/scripts/` that imports and wraps the web server. This prevents the web server's public `createWebServer` API from accumulating sidecar concerns, and makes the two personas explicitly visible in the file structure. The current code works correctly, but the implicit persona split makes it harder for future contributors to understand which code paths activate under which conditions.
