### 1. TUI app.ts Decomposition — The Recommended Extraction Was Executed
**Impact**: Positive — largest LOC decrease in TUI history

| `app-session-lifecycle.ts` | 191 | Session startup, thread resume/hydration, active-thread sync |
| `app-runtime-state.ts` | 80 | Mutable runtime state (turn timing, steering, thread IDs, mode) |

The net increase (+211 LOC across the system) comes from interface definitions and proper dependency injection — each extracted module has a typed `Deps` interface (e.g., `AppEventControllerDeps`, `AppSessionLifecycleDeps`). This is the correct extraction pattern: modules communicate through explicit interfaces, not through implicit shared class state.

The extraction is **stronger** than the simple class-internal split previously recommended. Instead of extracting methods within the same class, the developer created independent modules with typed dependency interfaces. This makes each module testable in isolation and prevents re-coupling.

## Forward Compatibility

### 16. CLI Test Infrastructure — Fragile Global Stdout Capture
**Horizon**: Breaks with any new test that writes to stdout during test execution

Five consecutive commits (772d758, 3d228f0, b8f9999, 46b4bb2, 18bfe90) hardened the same test infrastructure: global stdout listener capture in `tui-app.test.ts` and `runner.test.ts`. The pattern is:
1. Tests capture terminal output by intercepting stdout writes
2. Multiple tests share the stdout listener, causing race conditions
3. Each fix adds more isolation (per-test listeners, debounced assertions, filtering)

This incremental patching approach is fragile. The next test that exercises terminal output will likely encounter the same race. The root cause is that the test infrastructure couples to a global resource (stdout) rather than using a test-local terminal mock.

## Novel Perspective

### The Five-Commit Test Deflake Cluster Reveals an Architectural Missing Layer

- **Why no one has seen this**: Previous reviews tracked test quality metrics (coverage, flakiness) as a testing concern. The 5 consecutive test-fix commits were categorized as "CLI test stabilization" — a quality improvement, not an architectural finding. But the root cause is not test quality; it's a missing abstraction layer between the TUI framework and its test harness.

- **Evidence**:
  1. All 5 commits (772d758, 3d228f0, b8f9999, 46b4bb2, 18bfe90) modify the same 2 files: `tui-app.test.ts` and `runner.test.ts`. Each commit adds more isolation to stdout capture: per-test listeners, debounced assertion windows, filtered output matching, removed global cleanup hooks.

  2. The TUI framework (`packages/cli/src/tui/framework/`) has a `Terminal` class (terminal.ts), a `TUIRenderer` (renderer.ts), and a `Container` (container.ts). These abstract the terminal for production use. But tests bypass this abstraction and capture raw `process.stdout.write` calls — they test at the wrong layer.

  3. The web frontend doesn't have this problem. Web tests can mount React components with test renderers and assert on virtual DOM output. The TUI has no equivalent: there's no "test renderer" that captures the TUI's intended output without going through the actual terminal.

  4. The `tui-app.test.ts` file has grown to handle increasingly complex output assertions (delta streams, message rendering, tool blocks) while fighting stdout races. Each new TUI test will face the same infrastructure challenge.

  The pattern reveals a gap: **the TUI framework provides a production rendering layer (`Terminal` → `TUIRenderer` → `Container`) but no test rendering layer**. Web has React's test utilities. The TUI has `process.stdout.write` interception. This is not a "write better tests" problem — it's a "the framework doesn't support testability" problem.

- **Implication**: Before the next round of TUI feature tests, consider adding a `TestTerminal` or `TerminalCapture` class to `packages/cli/src/tui/framework/` that collects rendered output without touching `process.stdout`. Tests would instantiate `App` with a `TestTerminal` and assert on its captured output buffer. This eliminates the global stdout race entirely and makes every future TUI test simpler to write.

  The cost is ~50-80 LOC for the test terminal adapter. The payoff is eliminating the recurring 5-commit deflake cycles and making TUI testing as reliable as web component testing.
