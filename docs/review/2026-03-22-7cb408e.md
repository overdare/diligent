## Structural Integrity

### Core types.ts duplicates protocol data-model.ts — 82 LOC of structural types defined twice
**Impact**: Compounds over time

`packages/core/src/types.ts` defines `ContentBlock`, `TextBlock`, `ImageBlock`, `LocalImageBlock`, `ThinkingBlock`, `ToolCallBlock`, `UserMessage`, `AssistantMessage`, `ToolResultMessage`, `Message`, `StopReason`, and `Usage` as hand-written interfaces (lines 1–82). These same types exist as Zod-inferred types in `packages/protocol/src/data-model.ts` (lines 1–22, 24–73). Core does not import from protocol — it maintains its own parallel definitions.

This is the oldest type duplication in the project. It predates the protocol package: core's types were written first, and protocol's Zod schemas were modeled after them. Now that protocol is the canonical source of truth (per ARCHITECTURE.md), core's copies are legacy. Every consumer of `@diligent/core/types` (`packages/runtime/`, `packages/core/src/llm/types.ts`, `packages/core/src/agent/`) transitively depends on the duplicate.

Additionally, `ThinkingEffort` is duplicated as a string literal union in `packages/core/src/llm/types.ts:13` instead of being imported from protocol. Commit 23a6d03 fixed the `ThinkingEffortSchema` import in runtime config but did not address the type alias in core.

If the protocol `ContentBlock` union gains a new variant (e.g., `AudioBlock`), it must be manually added to `core/src/types.ts` or core consumers silently miss it. The risk is silent divergence — TypeScript won't flag the mismatch because the types are structurally typed and defined independently.

**Evidence**: `packages/core/src/types.ts:1-82`, `packages/protocol/src/data-model.ts:1-82`, `packages/core/src/llm/types.ts:13`

### Runtime approval and user-input types duplicated from protocol — 57 LOC
**Impact**: Compounds over time

`packages/runtime/src/approval/types.ts` (lines 3–10) defines `ApprovalRequest` and `ApprovalResponse` as interfaces. `packages/runtime/src/tools/user-input-types.ts` (lines 3–30) defines `UserInputOption`, `UserInputQuestion`, `UserInputSource`, `UserInputRequest`, and `UserInputResponse` as interfaces. Both files have zero imports from `@diligent/protocol`.

The protocol package defines identical Zod schemas for all of these types at `packages/protocol/src/data-model.ts:517-564`. The runtime versions are consumed by 12 import sites across `tools/`, `collab/`, `app-server/`, and re-exported from `packages/runtime/src/tools/index.ts:38`.

This duplication is more dangerous than the core types duplication because the runtime types cross the server boundary — they are serialized to JSON-RPC and deserialized by clients using protocol schemas. If runtime's `UserInputQuestion` interface drifts from protocol's `UserInputQuestionSchema`, the server sends data that fails client-side validation.

**Evidence**: `packages/runtime/src/approval/types.ts:3-10`, `packages/runtime/src/tools/user-input-types.ts:3-30`, `packages/protocol/src/data-model.ts:517-564`, 12 import sites across runtime

### render-payload.ts hardcodes 7 tool names as string comparisons
**Impact**: Compounds with each new tool

`packages/runtime/src/tools/render-payload.ts` (863 LOC) uses `normalizedToolName === "read"`, `"bash"`, `"apply_patch"`, `"write"`, `"update_knowledge"`, `"search_knowledge"`, and `"plan"` in `createToolStartRenderPayload()` and `createToolEndRenderPayloadFromInput()`. ## Cross-Package Coherence

### Type flow chain has two breaks: core/types.ts and runtime approval/user-input types
**Scope**: core, runtime, protocol

The intended type flow is `protocol → core → runtime → cli/web/desktop`. Two locations break this chain:

1. **core/src/types.ts**: Defines 11 types independently instead of importing from protocol. Core's `package.json` does not list `@diligent/protocol` as a dependency. This is likely the historical reason — core was written before protocol existed.

2. **runtime approval + user-input types**: 7 interfaces defined independently instead of importing from protocol. Runtime does depend on protocol (`package.json`), so there's no structural barrier — this is likely an oversight from when these types were created before the protocol package absorbed them.

Web and CLI correctly import from protocol. The break is in the middle of the chain (core and runtime), which means the single-source-of-truth guarantee exists only at the edges.

**Evidence**: `packages/core/package.json` (no `@diligent/protocol` dependency), `packages/runtime/package.json` (has `@diligent/protocol` dependency), import analysis showing 0 protocol imports in `approval/types.ts` and `user-input-types.ts`

### Plugin SDK re-exports are correctly sourced from protocol
**Scope**: plugin-sdk, protocol

Commit 9dc6cd8 added `packages/plugin-sdk/src/index.ts` which re-exports `ApprovalRequest`, `ApprovalResponse`, `UserInputRequest`, `UserInputResponse` from protocol. This is the correct pattern — external consumers get types from the canonical source.

However, the runtime (which the plugin-sdk wraps) uses its own duplicated types internally. If a plugin author compares the SDK types with the runtime types, they are structurally compatible today but not referentially identical. This hasn't caused issues yet but is a latent integration risk.

**Evidence**: `packages/plugin-sdk/src/index.ts`, `packages/runtime/src/tools/index.ts:38`

## Forward Compatibility

### Protocol version hardcoded at 1 with no negotiation — breaks on first protocol change
**Horizon**: Breaks when D089 (Thread Fork) or D091 (Thread Archive) ship protocol-breaking changes

`packages/runtime/src/app-server/server.ts` hardcodes `protocolVersion: 1` (commit e99eab7). There is no version negotiation handshake. When the protocol evolves (D089 Thread Fork requires new RPC methods), old clients connecting to a new server will receive unknown notifications with no graceful degradation path.

This is low-cost to address now (add a capabilities exchange during connection setup) but becomes a breaking change to fix once multiple client versions exist in the field.

**Evidence**: `packages/runtime/src/app-server/server.ts`, commit e99eab7, D089/D091 in decisions.md

### Tool name strings scattered across 4 files — adding a tool requires manual sync
**Horizon**: Breaks at scale when plugin tools need custom rendering or mode filtering

Tool names appear as string literals in:
1. `tools/immutable.ts` — 3 names in `IMMUTABLE_TOOLS`
2. `agent/mode.ts` — 7 names in `PLAN_MODE_ALLOWED_TOOLS`
3. `collab/registry.ts` — 4 names in `COLLAB_TOOL_NAMES`
4. `tools/render-payload.ts` — 7 names in if-else chains

Each new built-in tool must be manually considered for all 4 locations. Currently there are 17 built-in tools; at 25+ this manual sync becomes a meaningful source of bugs. A tool metadata object (name + capabilities like `immutable`, `planModeAllowed`, `renderStrategy`) would centralize this.

**Evidence**: `packages/runtime/src/tools/immutable.ts:4`, `packages/runtime/src/agent/mode.ts:14-22`, `packages/runtime/src/collab/registry.ts`, `packages/runtime/src/tools/render-payload.ts`

## Novel Perspective

### The core package cannot depend on protocol — and this is the root cause of all type duplication

**Why no one has seen this**: Previous reviews flagged individual type duplications (ThinkingEffort, content blocks, approval types) as independent findings. The common root cause — that `packages/core/` has no dependency on `@diligent/protocol/` — was never identified as the structural constraint driving all of them.

**Evidence**: `packages/core/package.json` lists `@anthropic-ai/sdk`, `@google/genai`, `openai`, `jsonc-parser`, and `zod` as dependencies. `@diligent/protocol` is absent. Core's `types.ts` was written first (it's the original type source), and protocol's Zod schemas were modeled after it. The dependency arrow was never reversed.

Meanwhile, runtime depends on both core and protocol. It re-exports core types to its consumers (tools, collab, app-server) but also uses protocol types for RPC serialization. This creates a two-source situation: runtime's internal types come from core's duplicates, while its RPC types come from protocol's originals.

The fix is to add `@diligent/protocol` as a dependency of core, then replace core's `types.ts` interfaces with re-exports from protocol. This would:
- Eliminate 82 LOC of duplicate type definitions in core
- Make `ThinkingEffort` import work naturally (core imports from protocol, not redefining)
- Unblock runtime from importing approval/user-input types from protocol (runtime currently imports these from its own files because core set the precedent of not depending on protocol)

The risk is low: protocol has zero dependencies beyond `zod`, so adding it to core creates no circular dependency. The compile-time parity check in `llm/provider-name.typecheck.ts` already proves the types are structurally identical — the migration is mechanical.

**Implication**: Instead of fixing each type duplication individually (as previous reviews recommended), fix the dependency graph once. Add `@diligent/protocol` to core's dependencies, replace `core/src/types.ts` with re-exports, then update runtime's approval and user-input types in the same pass. One session, ~30 minutes, eliminates the entire class of findings.
