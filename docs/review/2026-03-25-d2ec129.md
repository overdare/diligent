## Novel Perspective

### Automated Remediation PRs Lack Merge-Persistence Verification — the Pipeline Has No Post-Merge Guard

**Why no one has seen this**: The automated remediation pipeline (passion-junior → PR → merge) has operated flawlessly for 4+ review cycles with remediation rates of 75-100%. The pipeline was designed to land fixes, and it does. What it was never designed to do — and what was never needed until now — is verify that a merged fix *stays* fixed across subsequent commits.

**Evidence**:

1. **PR #50** (a1260db) correctly changed `frontmatter.ts:21-23` from a hard error to a `console.warn`. The PR was reviewed, merged, and the review cycle recorded it as resolved.

2. **Commit 1168ab7** modified the same file as part of an overdare feature commit. The `git diff` shows the warning was replaced back with the hard error. This commit touched 2 files: `instance.params.ts` (the intended change, +511/-101) and `frontmatter.ts` (-4/+1, the accidental revert).

3. **The test suite provides no protection**: `frontmatter.test.ts:42-56` tests that unknown tools produce an error. PR #50 should have changed this test to expect a warning instead. Either the test was not updated (a bug in the PR) or it was updated and then reverted along with the code change. Either way, CI passes with the regression present.

4. **The review pipeline has no post-merge invariant check**: Today's review detected the regression through manual git diff analysis. In a longer commit range (e.g., 20+ commits between reviews), this kind of silent revert could easily go undetected.

5. **Root cause hypothesis**: The 1168ab7 commit appears to have been authored on a branch that was based on a commit *before* a1260db merged. When the branch was merged or rebased, the frontmatter.ts changes from PR #50 were resolved in favor of the branch version (the old code), not the mainline version.

**Implication**: The review→remediate→verify cycle needs one additional step: after automated PRs merge, their changes should be tested for persistence. Options:
- (a) Add a post-merge CI check that diffs merged PR files against HEAD and flags reversions
- (b) Update `frontmatter.test.ts` to test *both* behaviors (the warning path) so that any reversion breaks CI
- (c) Accept the risk and rely on the next tech-lead review to catch it (current implicit strategy)

Option (b) is the minimum viable fix: PR #50's test should have asserted the warning behavior, making the regression a CI failure. This is a general lesson — every automated remediation PR should include a test that would fail if the fix were reverted.
