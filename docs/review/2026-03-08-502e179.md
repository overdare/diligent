## Novel Perspective

### The Extraction Debt Illusion: Duplicated Helpers as a Hidden Coupling Layer

- **Why no one has seen this**: Previous reviews tracked large-file complexity as the primary decomposition risk. Once a file is "extracted," it's marked as resolved. But the extraction process itself introduced a new kind of coupling that doesn't show up in any metric tracked so far — **cross-file helper function duplication**.

- **Evidence**: The thread-hydration extraction (19da8ab) created 6 duplicated helper functions across `thread-store.ts` and `thread-hydration.ts` (see Finding #15). But this pattern already existed before: the `COLLAB_RENDERED_TOOLS` set is defined identically at `thread-store.ts:18` and `thread-hydration.ts:8`. The `zeroUsage` constant appears in both files. The `server.ts` decomposition is cleaner — it uses the `ThreadHandlersContext` interface pattern instead of duplicating internals — but it introduced its own coupling: `thread-handlers.ts` re-imports `ThreadRuntime` from its own type definition, which `server.ts` also imports. The pattern reveals a systematic gap: **the project has no shared utility layer within the web client package**. `packages/web/src/client/lib/` contains self-contained modules (thread-store, rpc-client, hooks) but no `thread-utils.ts` for shared helpers. Every extraction creates pressure to either duplicate or restructure.

- **Implication**: As more extractions happen (the natural trajectory for a growing codebase), this pattern compounds. Each extraction that copies-instead-of-shares creates a sync point. The fix is lightweight: introduce a `thread-utils.ts` in `packages/web/src/client/lib/` for shared state manipulation helpers (`addSeen`, `withItem`, `updateItem`, `stringifyUnknown`, `extractUserTextAndImages`, `parsePlanOutput`, `COLLAB_RENDERED_TOOLS`, `zeroUsage`). This should be done now, before the next extraction (D093 streaming delta reducers), which will likely need the same helpers in a third location.
