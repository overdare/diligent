## Forward Compatibility

### Per-registry maxAgents creates unbounded global agent count when nesting is enabled
**Horizon**: Breaks when nested subagents are used at depth > 2

Each `AgentRegistry` instance maintains its own `maxAgents` counter (registry.ts:94). When `allowNestedAgents=true`, `buildDefaultTools` creates a fresh `AgentRegistry` for the child (defaults.ts:77-86). That child registry has its own independent `maxAgents` limit. The total concurrent agent count across the tree is bounded by `maxAgents^depth`, not `maxAgents`.

With default `maxAgents=5`: a 3-level deep chain could create 5+25+125=155 concurrent sessions, each with its own `SessionManager`, stream function, and in-memory state. There is no global agent counter across registry instances.

In practice, the default `allowNestedAgents=false` prevents this. But when explicitly enabled, there's no structural depth limit — only the LLM's compliance with the prompt policy "use only if the parent instruction clearly requires further delegation."

**Evidence**: `packages/runtime/src/collab/registry.ts:94-97` (per-instance limit), `packages/runtime/src/tools/defaults.ts:77-86` (child registry creation), `packages/runtime/src/collab/spawn-agent.ts:66` (opt-in flag)

## Novel Perspective

### The Double-Exclusion Pattern in Child Tool Filtering Masks a Latent Ordering Dependency

**Why no one has seen this**: The nested subagent feature (ed18abd) was just committed, and the child tool filtering cascade has been reviewed only as "each stage can only restrict, never expand." The filter's correctness is obvious for the *current* code. The novel angle is that the filtering contains a **redundant exclusion** that masks a dangerous ordering dependency — one that will manifest if the filtering logic is ever reordered or refactored.

**Evidence**:

1. **Line 110–113** conditionally removes collab tool names from `allowedChildToolNames` when `allowNestedAgents=false`.

2. **Line 128** unconditionally removes collab tools from `childTools`: `!COLLAB_TOOL_NAMES.has(tool.name) && allowedChildToolNames.has(tool.name)`.

3. **Line 192** applies a *third* filter after `buildDefaultTools`: `result.tools.filter((tool) => allowedChildToolNames.has(tool.name))`.

When `allowNestedAgents=false`: collab tools are removed at lines 110–113, then again at line 128, then can't survive line 192 either. Three redundant removals.

When `allowNestedAgents=true`: collab tools stay in `allowedChildToolNames` (skip 110–113), get removed from `childTools` at line 128 (unconditionally), then get re-added by `buildDefaultTools` (which creates fresh collab tools via `createCollabTools`), then survive line 192 (because they're in `allowedChildToolNames`).

4. **The latent ordering dependency**: The unconditional collab exclusion at line 128 exists because child agents need *fresh* collab tools (with their own registry), not inherited parent tools. But this reason is not documented — it looks like a redundant filter. If a future refactorer removes the "redundant" line 128 exclusion (since lines 110–113 already handle it for the default case), the `allowNestedAgents=true` path breaks: children would get *both* parent collab tools (stale, pointing to parent's registry) AND fresh collab tools from `buildDefaultTools`, resulting in duplicate tool names.

5. **The `allowedChildToolNames` Set serves dual purpose**: For regular tools, it's a filter (lines 120–127). For collab tools, it's purely a signal — its only effect is setting `nestedCollabEnabled` (line 131) and surviving the post-buildDefaultTools filter (line 192). This dual purpose is invisible without reading all 50 lines of the cascade together.

**Implication**: The tool filtering cascade needs a clarifying comment at line 128 explaining *why* collab tools are unconditionally excluded from `childTools` even when `allowedChildToolNames` includes them. Better yet, extract the 5-stage filter into a named function (`resolveChildToolAccess()`) that documents the three concerns separately: (1) which non-collab tools survive, (2) whether collab tools should be re-created, (3) the post-build reconciliation at line 192. This prevents a future refactorer from "simplifying" away a load-bearing redundancy.
