## Forward Compatibility

### Provider Count Scaling
**Horizon**: Breaks when provider count exceeds ~10

Current provider addition is clean, but scaling signals to watch:
- `ProviderNameSchema` is a static `z.enum` — every new provider requires a protocol package change
- `PROVIDER_FACTORIES` in `provider-manager.ts` is a static dictionary — same constraint
- Auth store has a flat key-per-provider structure

At 6 providers this is fine. At 10+ it becomes a maintenance burden. The eventual fix is a provider registry that supports dynamic registration (similar to how tools/plugins work). Not urgent.

### Web Client Complexity Ceiling
**Horizon**: Breaks when next major UI feature requires cross-cutting state changes

Web client has several large files approaching complexity thresholds:
- `App.tsx` (729 LOC) — orchestrates state, RPC, modals, thread dispatch
- `thread-store.ts` (712 LOC) — protocol reducer + view normalization
- `InputDock.tsx` (607 LOC) — textarea, commands, model/effort controls
- `use-app-actions.ts` (543 LOC) — action handlers

These files are functional and well-typed, but a feature like thread fork or multi-panel views would likely push `App.tsx` past 900 LOC and require touching all four files simultaneously. Consider extracting:
- State management into a dedicated context/store (React Context or Zustand)
- `InputDock` sub-components: `<SlashCommandAutocomplete>`, `<ModelSelector>`, `<UsageTray>`

## Novel Perspective

### The OpenAI-Compatible Abstraction Is Becoming a Shadow Standard

- **Why no one has seen this**: Previous reviews focused on individual provider files and registry consistency. The cross-cutting role of `openai-compatible.ts` as an implicit provider contract hasn't been examined.

- **Evidence**: Three providers now share the same message/tool building helpers:
  - `zai.ts:8-10` imports `buildOpenAICompatibleMessages`, `buildOpenAICompatibleTools`, `handleChatCompletionsEvents` from `openai-compatible.ts`
  - `vertex.ts` uses the same pattern
  - `openai.ts` uses `openai-shared.ts` (the heavyweight sibling)

  This means `openai-compatible.ts` is effectively the "provider implementation contract" — any breaking change to its signatures affects 3+ providers simultaneously. Yet it has no explicit interface/type contract, no dedicated tests, and no documentation of its stability guarantees.

- **Implication**: The project should treat `openai-compatible.ts` as a semi-public internal API:
  1. Add explicit type signatures for `buildOpenAICompatibleMessages()` and `handleChatCompletionsEvents()` return types
  2. Add unit tests that verify the contract independently of any specific provider
  3. Document in a code comment or decision that this file is the reuse point for OpenAI-format providers

  If a future provider needs slight variations (e.g., different tool_choice format, non-standard streaming), the current approach of importing raw helper functions won't scale. A `createOpenAICompatibleStream(config: OpenAICompatibleConfig)` factory would be more resilient than requiring each provider to wire 3 separate imports.
