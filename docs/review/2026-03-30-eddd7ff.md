## Forward Compatibility

### Desktop dual-build scripts — package.ts and build-exe-only.ts
**Horizon**: Breaks when packaging logic diverges

`apps/desktop/scripts/package.ts` (493 LOC) and `apps/desktop/scripts/build-exe-only.ts` (132 LOC) are two independent packaging scripts. Both perform: version injection → web frontend build → sidecar compile → Tauri build → artifact collection. `build-exe-only.ts` is a simplified single-platform path for dev/testing that skips runtime bundle assembly.

The scripts share `lib/manifest.ts`, `lib/platforms.ts`, and `lib/version.ts`, but the core build sequences are duplicated. Today the duplication is manageable (132 vs 493 LOC), but if packaging logic evolves (new artifact types, new platform targets, new build steps), both scripts must be updated in sync.

**Evidence**: `apps/desktop/scripts/package.ts:1-493`, `apps/desktop/scripts/build-exe-only.ts:1-132`

### Desktop JSONC parser is a simplified subset
**Horizon**: Breaks when config.jsonc uses JSONC features beyond single-line comments

`apps/desktop/src-tauri/src/update.rs:89-101` implements a simple JSONC parser that strips `//` comments before JSON parsing. This handles the common case but fails on:
- `//` inside quoted strings (e.g., `"url": "https://example.com"` would be truncated)
- Block comments (`/* */`)
- Trailing commas

The current `config.jsonc` is simple enough that this works, and the only field read is `updateMode`. However, if the config evolves to contain URLs or complex values, the naive comment stripping could silently corrupt the parsed result.

**Evidence**: `apps/desktop/src-tauri/src/update.rs:89-101`

## Novel Perspective

### The Desktop Update System Has Created an Implicit Runtime Contract That No Package Owns

**Why no one has seen this**: Previous reviews focused on the TypeScript package dependency chain (`protocol → core → runtime → cli/web`). The desktop auto-update system introduces a new dependency surface that is entirely file-system-based and crosses the Rust/TypeScript boundary without any shared type definitions.

**Evidence**:

1. **The update manifest is a contract with no schema owner**: `apps/desktop/scripts/lib/manifest.ts:6-13` generates the `update-manifest.json` with a `{ version, platforms: { [platform]: { url, sha256, size } } }` structure. `apps/desktop/src-tauri/src/update.rs:23-36` deserializes this with `UpdateManifest` / `PlatformBundle` Rust structs. These two type definitions exist in different languages with no shared schema validation. If the TypeScript generator adds a field (e.g., `minShellVersion` for compatibility checks), the Rust consumer silently ignores it via `serde(default)`. If a required field is renamed, the Rust parser fails at runtime with a deserialization error.

2. **The runtime directory layout is a contract with no specification**: The update system produces a specific directory layout (`runtime/diligent-web-server`, `runtime/dist/client/`, `runtime/rg`, `runtime/defaults/`, `runtime/version.json`). Both `sidecar.rs` (lines for path resolution) and `init.rs` (defaults deployment) hardcode these path expectations. The packaging script (`package.ts:assembleRuntimeBundle`) produces this layout. If any of these three points of production/consumption change the expected layout, the others break silently — the sidecar binary won't be found, or defaults won't be deployed, or the health check will fail.

3. **The `version.json` installed-version metadata is written by Rust, potentially consumed by TypeScript**: `update.rs:414-424` writes `{ version, applied_at, sha256 }` to `runtime/version.json`. The sidecar (TypeScript) could read this to report its version, but currently doesn't. If it ever does, this is another cross-language schema with no shared definition.

4. **The platform identifier strings are a shared constant with no single source**: `current_platform()` in `update.rs:66-77` returns strings like `"darwin-arm64"`, `"windows-x64"`. `platforms.ts` in desktop scripts defines the same strings for manifest generation. These must match exactly for updates to resolve, but there's no compile-time or test-time validation that they agree.

5. **The precedent matters**: Today's layout is simple and managed by a single developer. But the update system is designed for production — SHA256 verification, atomic staging, fallback chains. As more artifacts enter the runtime bundle (additional plugins, model configs, skill files), the implicit layout contract grows without a formal schema. Each new artifact added to the bundle requires synchronized changes in: `package.ts` (production), `update.rs` (extraction), and `sidecar.rs`/`init.rs` (consumption).

**Implication**: Define the runtime bundle layout as a documented contract. A simple Markdown table in P059 or a new decision listing: (a) expected directory entries, (b) platform identifier mapping, (c) `version.json` schema, and (d) `update-manifest.json` schema. This is not over-engineering — it's documenting what already exists implicitly across 4 files in 2 languages so that future changes to any one file can be validated against the others.
