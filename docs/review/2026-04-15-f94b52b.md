## Forward Compatibility

### Web Reducer Adoption Window
**Horizon**: Blocks full reducer convergence; two review cycles without progress

The shared reducer is architecturally ready (133 LOC, 19 event types, test suite, collab coverage). Web adoption requires:
1. Delegate base `ConversationLiveState` fields to `applyAgentEvents()`
2. Keep Web-specific extensions (RenderItem mapping, thinking-duration, content-block finalization) as a thin layer composing on top

Risk of this remaining open: Web and protocol reducers continue to evolve independently. Any event-type additions that land only in the protocol reducer will create silent behavior gaps in the Web client until adoption happens. Conversely, Web-only event handling added to `thread-store.ts` widens the adoption delta.

### data-model.ts Approaching Extraction Threshold
**Horizon**: Compounds as event types and content blocks grow

At 695 LOC, `data-model.ts` contains 25 `AgentEvent` subtypes, 9 content block types, 10 tool render block types, collab types, metadata types, and domain models. While well-organized with Zod discriminated unions, it is becoming a navigation burden. The file serves as a single import point, which is convenient, but any new event type or content block adds to a growing linear scan for maintainers.

Natural split points exist:
- Event schemas (~160 LOC) → `events.ts`
- Content block schemas (~80 LOC) → `content-blocks.ts`
- Tool render schemas (~100 LOC) → `tool-render.ts`
- The remaining ~350 LOC (messages, metadata, domain models) stays in `data-model.ts`

This is not urgent but should be considered before the next batch of event-type additions.

### server.ts State Map Proliferation
**Horizon**: Compounds if per-cwd or per-session features continue landing in server.ts

server.ts now holds 4+ private Maps for state management:
- `lastUsedModelByCwd`, `lastUsedEffortByCwd` (model/effort caching)
- Connection, thread, subscription tracking Maps
- Debug-state Maps delegated to manager.ts

Each is individually reasonable, but the file's dual role (request dispatcher + state container) creates a growth attractor. The request dispatch extraction (previous review Action #4) would clarify server.ts's identity as primarily a state/lifecycle manager.

## Novel Perspective

### Plugin Hooks Are the Untested Half of the Plugin SDK

- **Why no one has seen this**: Previous reviews focused on tool execution as the plugin SDK's primary contract, because the Overdare plugins (studiorpc, rag, validator, analytics) are all tool-oriented. The `PluginHookInput`/`PluginHookResult` lifecycle hook interface (`UserPromptSubmit`, `Stop`) is equally prominent in the SDK but has zero known consumers.
- **Evidence**:
  - `packages/plugin-sdk/src/index.ts` exports both `Tool` (lines 5–14) and `PluginHookInput`/`PluginHookResult` (lines 23–40)
  - `apps/overdare-agent/plugins/` contains 4 plugins: all define tools, none implement hooks
  - `thirdparty/examples/external-tool-plugin/` (the example plugin) defines tools, not hooks
  - `packages/e2e/` has no test exercising the plugin hook dispatch path
  - Hook dispatch lives in `packages/runtime/src/hooks/` but is only tested implicitly through agent loop integration
- **Implication**: The hook contract is a published API with no consumer validation. If a plugin author builds on `PluginHookInput.stop_hook_active` or `PluginHookResult.blocked`, there is no guarantee the runtime's hook dispatch behaves as the SDK types promise. Before promoting plugin hooks to external authors, either:
  1. Add an e2e test that installs a hook-implementing plugin and verifies `UserPromptSubmit` blocking + `Stop` hook execution
  2. Or add a dedicated hook example to `thirdparty/examples/`

  This is especially important because hooks have **blocking semantics** (a hook can block a user prompt) — unlike tools, a broken hook can silently prevent user interaction rather than just producing a wrong result.
