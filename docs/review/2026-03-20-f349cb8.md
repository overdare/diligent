## Novel Perspective

### The Web and CLI State Management Are Diverging Into Incompatible Paradigms — And The Gap Will Widen

- **Why no one has seen this**: Previous reviews tracked file sizes and duplication. They noted that the web uses a functional reducer pattern (thread-store.ts → `reduceAgentEvent()`) while the CLI uses an imperative class pattern (transcript-store.ts → `TranscriptStore`). But no review has identified that these two paradigms create fundamentally different extensibility constraints, and that the web's decomposition success this period is structurally impossible for the CLI.

- **Evidence**:
  1. **Web's extraction was clean because reducers compose**: `thread-store.ts` was a pure function: `(state, event) → state`. Extracting `collab-reducer.ts` and `tool-reducer.ts` was straightforward — each is a pure function `(state, event) → state` that the main reducer delegates to. The sub-reducers have zero side effects, zero internal state, and can be tested in isolation with simple state-in/state-out assertions. The test file (`thread-store.test.ts`, 1,857 LOC) proves this composability.

  2. **CLI's extraction is blocked because classes don't compose the same way**: `transcript-store.ts` (983 LOC) is a `TranscriptStore` class with 15+ private fields (`items`, `busyStartedAt`, `thinkingMarkdown`, `overlayStatus`, `statusBlinkTimer`, `statusRefreshTimer`, etc.), timer intervals (`ensureStatusTimers`, `cleanupStatusTimersIfIdle`), and methods that mutate internal state then call `this.options.requestRender()`. Extracting a sub-handler requires either: (a) passing the entire `TranscriptStore` instance to the handler (exposing internals), (b) creating a shared state interface (complex), or (c) refactoring to a reducer pattern (large rewrite).

  3. **The paradigm gap is already visible in the collab feature**: The web's collab rendering lives in `collab-reducer.ts` (283 LOC) — a pure function imported by thread-store. The CLI's collab rendering lives inline in transcript-store.ts as private methods. When the next collab feature lands, web adds a case to a pure function; CLI adds methods and potentially new private fields to an already-large class. The growth patterns are asymmetric.

  4. **The CLI's transcript-store.test.ts (490 LOC) vs web's thread-store.test.ts (1,857 LOC) reveals the testing gap**: The CLI test file is 3.8x smaller despite the CLI source file being 65% larger (983 vs 597 LOC). This isn't just a coverage gap — it reflects the difficulty of testing imperative class state vs. pure reducer functions. The web's tests construct a state object, call the reducer, and assert the result. The CLI's tests must instantiate the class with mock options, call methods in sequence, and inspect accumulated side effects.

  5. **The divergence will compound**: Every new event type or rendering feature requires changes in both files. But the web change is "add a case to a sub-reducer" (10-30 LOC, testable in isolation) while the CLI change is "add methods to the class, manage new private state, coordinate with timer lifecycle" (20-60 LOC, testable only via the class instance). Over 10 more features, the web's modular structure absorbs growth gracefully while the CLI's monolithic class accumulates friction.

- **Implication**: The CLI's `TranscriptStore` should be evaluated for a paradigm shift — not just extraction, but a move toward a reducer-like pattern where event processing is separated from rendering state management. This doesn't require abandoning the class entirely: the class can own timers and render-triggering while delegating event-to-state-change logic to pure functions. The pattern:

  ```typescript
  // Pure function — extractable, testable, composable
  function reduceTranscriptEvent(state: TranscriptState, event: AgentEvent): TranscriptState { ... }

  // Class owns lifecycle — timers, render triggers, input state
  class TranscriptStore {
    private state: TranscriptState;
    handleEvent(event: AgentEvent) {
      this.state = reduceTranscriptEvent(this.state, event);
      this.requestRender();
    }
  }
  ```

  This would give the CLI the same extraction and testing benefits the web gained this period, without a full rewrite. The cost is ~2-4 hours of refactoring. The payoff is preventing the CLI from becoming the project's next structural bottleneck as collab features continue to land.
